## 第7条(1)① — 求人情報の取得・利用について

### ア: 職業紹介と関係がない目的での利用禁止

**判定: ⚠️ 要確認**

- worker.js内のEXTERNAL_JOBSは、LINE Botでの求職者マッチングに使用されている（職業紹介目的）。
- しかし、`lp/job-seeker/index.html` のフッターに「施設データ出典: ハローワーク求人」の記載がある。LP自体はマーケティング目的であり、求人データを「集客のエサ」として利用している側面がある。
- `lp/job-seeker/jobs-summary.json` にはハローワーク求人から抽出したと思われるサンプル求人（職種名・給与・休日）が含まれ、診断機能（shindan.js）で公開表示されている。

**是正案:**
- jobs-summary.jsonのsample/blurredデータがハローワーク求人由来であれば、職業紹介目的以外の利用に該当する可能性がある。公開Webサイトでの集客利用は規約の趣旨に反する。
- 診断結果のサンプル求人は、自社受理済み求人のみに限定するか、統計情報（件数・給与レンジ）のみの表示にする。

---

### イ: 第三者・不特定多数への提供禁止

**判定: 🔴 違反リスク（最重要）**

規約原文:
> 「求人対象団体以外の第三者（求人対象団体の求職者を除く）への提供及びインターネット等で求職者以外の不特定多数の者への提供は行わないこと。」

**現状の問題点:**

1. **worker.js内のEXTERNAL_JOBSにハローワーク求人が直接埋め込まれている**
   - worker.jsはCloudflare Workerとしてデプロイされるが、ソースコード自体はGitHubパブリックリポジトリ（Quads-Inc/robby-the-match）に公開されている。**（`gh repo view` で `isPrivate: false, visibility: PUBLIC` を確認済み）**
   - EXTERNAL_JOBSには事業所名・職種・給与・勤務地・仕事内容等がハードコードされており、GitHubリポジトリを閲覧すれば誰でもアクセスできる。

2. **LINE Botでのマッチング結果表示は規約内**
   - LINE Botに友だち追加し、ヒアリングに回答した求職者にのみ求人カードを表示する仕組み。
   - これは「求人対象団体の求職者」への提供であり、規約適合。

3. **公開サイトでのサンプル求人表示（jobs-summary.json + shindan.js）**
   - 診断結果画面で「訪問看護ステーション 管理者候補 月給34〜40万円」等のサンプル求人が表示される。
   - これらはblurred（ぼかし）処理されているが、事業所名こそないものの職種名・給与等の求人内容が認証なしで誰でも閲覧可能。
   - 不特定多数への提供に該当する可能性が高い。

4. **salary-map.htmlの施設データ**
   - salary-map.htmlはFACILITIES変数で施設データを表示しているが、これは厚労省病床機能報告由来のデータであり、ハローワーク求人データとは別。直接の規約違反ではない。

**是正案（緊急度: 高）:**
- GitHubリポジトリをプライベートに変更する（最優先）。または worker.js のEXTERNAL_JOBSをD1データベースに完全移行し、ソースコードから求人データを除去する。
- jobs-summary.jsonからハローワーク由来のサンプル求人データを削除し、統計値（件数・給与レンジ）のみに置き換える。
- 公開サイト上では「○○件の求人あり」等の件数表示にとどめ、具体的な求人内容はLINE Bot内（認証済み求職者のみ）で表示する。

---

### エ: ハローワーク由来であることの明示 + 職業紹介手続きの明示

**判定: 🔴 違反リスク**

規約原文:
> 「安定所から情報提供を受けた求人情報であること及び求職者が職業紹介を希望する際の手続き等を明示すること。」

**現状の問題点:**
- LINE Botで表示される求人カード（buildFacilityFlexBubble関数）には「ハローワーク（安定所）から提供を受けた求人情報である」旨の表示が一切ない。
- ランクラベル（「特におすすめ」「おすすめ」「好条件」）は表示されているが、情報源の明示がない。
- 求職者が職業紹介を希望する際の手続き説明もない。

**是正案:**
- Flex Messageのフッターまたはボディに「出典: ハローワーク求人情報提供サービス」を追加する。
- 「この求人が気になる」ボタン押下後のフローで、職業紹介の手続き（求職申込み → 求人受理 → 紹介 → 面接）を説明するメッセージを追加する。

---

### オ: 求人内容の正確な引用・改変禁止

**判定: 🔴 違反リスク**

**現状の問題点:**
- `hellowork_rank.py` でスコアリング（S/A/B/C/Dランク）を付与している。これ自体は求人内容の改変ではないが、「特におすすめ」「おすすめ」等のラベルは元の求人にない評価情報の付加であり、規約クの問題と重なる。
- `hellowork_to_jobs.py` で仕事内容を150文字に切り詰めている（`desc[:147] + "..."`）。元の求人内容を省略しており、「正確な引用」に反する可能性がある。
- 給与表示を「月給○万円」形式に変換している。元データの正確な金額表現と異なる可能性がある。

**是正案:**
- 仕事内容の切り詰めは避けるか、「続きはハローワークでご確認ください」と明示する。
- 給与はAPIから取得した原文をそのまま表示するか、変換後の表現が原文と異なる場合はその旨を明示する。

---

### カ: 最新情報の提供

**判定: ✅ 適合**

- `hellowork_fetch.py` は毎朝06:30にcronで実行されており、最新データを日次で取得している。
- `hellowork_diff.py` で日次差分を検出し、新着・終了求人を追跡している。
- 有効期限（kjyukoymd）も取得されており、期限切れ求人の管理が可能な設計。

---

### キ: 職業安定法第5条の3に規定する労働条件等の省略禁止

**判定: 🔴 違反リスク**

職業安定法第5条の3で明示が義務付けられている項目:
1. 業務内容
2. 契約期間
3. 就業場所
4. 始業・終業時刻/休日/時間外労働
5. 賃金
6. 社会保険・労働保険の適用

**現状の問題点:**
- worker.jsのEXTERNAL_JOBSに含まれるフィールド: n(事業所名), t(職種), sal(給与), sta(最寄り駅), hol(年間休日), bon(賞与), emp(雇用形態), wel(福利厚生), desc(仕事内容80字), loc(勤務地), shift(勤務時間)
- **契約期間**が含まれていない。
- **社会保険・労働保険の適用**が含まれていない。
- **始業・終業時刻**はshift1/shift2から変換されているが、省略されるケースがある。
- desc(仕事内容)は150文字に切り詰められており、原文の完全性が損なわれている。

**是正案:**
- EXTERNAL_JOBSの構造に契約期間・社会保険情報を追加する。
- hellowork_fetch.pyは既にinsurance（加入保険）を取得しているので、hellowork_to_jobs.pyの変換で含める。
- 契約期間のフィールド（hellowork_fetch.pyへの追加が必要）も取得・表示する。

---

### ク: 独自情報の付加に関するルール

**判定: 🔴 違反リスク**

規約原文:
> 「求人対象団体が提供された求人に独自に情報を付加する場合は、関係法令の規定を遵守し、求人対象団体が求人事業主の同意を得るとともに、付加した情報は求人対象団体の責任で付加したことを求人事業主及び求職者に明確に伝えること。」

**現状の問題点:**
- ランク（S/A/B/C/D）とラベル（「特におすすめ」「おすすめ」「好条件」）は独自に付加した評価情報。
- 推薦理由バッジ（`job.reasons`）も独自付加情報。
- これらが「求人対象団体の責任で付加した」旨の表示がない。
- 求人事業主の同意を得ていない。

**是正案:**
- ランク・推薦理由が独自評価であることを求職者に明示する（例:「※ランク・推薦理由はナースロビー独自の評価です」）。
- 求人事業主への初回連絡時に、独自ランク付けの同意を書面で取得する運用を追加する。

## 第7条(1)② — 職業紹介について

### ア: 手数料の十分な説明

**判定: ⚠️ 要確認**

規約原文:
> 「安定所に求人を申し込んだ求人事業主は、そもそも無料の職業紹介を受けることを希望していたものであり、有料職業紹介事業者は、手数料について十分に説明すること。」

**現状の問題点:**
- worker.js内のシステムプロンプトに「手数料は求人側負担、求職者は完全無料であることを伝える」とあるが、これは求職者向けの説明。
- 求人事業主への手数料説明（「元々無料のハローワーク求人だった → 当社経由では10%の手数料が発生する」）の運用プロセスが不明。

**是正案:**
- 求人事業主への初回連絡テンプレートに、ハローワーク由来の求人であること・手数料10%の説明を明記する。
- 説明記録を残す運用（メール/書面）を整備する。

---

### イ: 求人受理後の書面明示（6項目）

**判定: ⚠️ 要確認**

必須明示項目:
- (ア) 取扱職種の範囲
- (イ) 手数料に関する事項
- (ウ) 苦情の処理に関する事項
- (エ) 求人者の情報の取扱い
- (オ) 求職者の個人情報の取扱い
- (カ) 返還金制度に関する事項

**現状の問題点:**
- これらの書面テンプレートの存在がコードベース上確認できない。
- proposal.html（提案書）には手数料の記載はあるが、上記6項目を網羅しているか未確認。

**是正案:**
- 求人受理時の書面テンプレートを作成し、6項目を全て網羅する。
- 書面交付の運用フローを確立する。

---

### ク: 営業活動の制限

**判定: ✅ 適合**

- 現状、求人事業主へのアウトバウンドセールスは行われていない（事業立ち上げ初期のため）。
- 社長指示で「A病院の求人はある。紹介すれば契約できる」とあり、特定の既存関係先のみとの取引が中心。
- 大量の求人事業主に一斉営業する仕組みは構築されていない。

## 第19条 — 利用状況の報告

### 年次報告（4月20日まで）

**判定: ⚠️ 要確認**

規約原文:
> 「求人対象団体は、提供された求人情報をもとに採用が決定した人数等の利用状況を年度毎にとりまとめ、各年度の最終月（4月）の翌月（4月20日まで）に報告すること。有料職業紹介事業者については、手数料の合計額も記入すること。」

**現状の問題点:**
- 年次報告の仕組み（スクリプト・リマインダー）がコードベース上に存在しない。
- 本日が2026-04-14であり、報告期限の4月20日まで**残り6日**。
- 採用決定人数が0人でも報告は必須（規約注4）。
- 未報告の場合、期限翌日から1か月経過でサービス6か月停止→さらに未改善で利用解除。

**是正案（緊急度: 最高）:**
- 直ちに管轄労働局に利用状況を報告する。求人・求職情報提供サービスサイトから報告可能。
- 採用決定人数・手数料合計額を集計する（0件でも報告必須）。
- 来年度以降のリマインダーをcronまたはカレンダーに登録する。

## 技術的是正の具体案

### 1. EXTERNAL_JOBSのD1完全移行（第7条①イ対応）

```
現状: worker.js にハードコード → GitHubで公開
改善: D1 jobsテーブルに全件格納 → worker.jsからデータ削除
     hellowork_to_d1.py は既に存在するため、パイプラインは整っている
```

### 2. LINE Bot求人カードの修正（第7条①エ・ク対応）

buildFacilityFlexBubble() のfooterに以下を追加:
```
「求人情報出典: ハローワーク求人情報提供サービス」
「※ランク評価はナースロビー独自の判断です」
```

### 3. jobs-summary.jsonの無害化（第7条①イ対応）

sample/blurredから具体的な求人情報を削除:
```json
{
  "yokohama_kawasaki": {
    "count": 218,
    "salary_range": "月給24〜45万円",
    "holidays_avg": 118
  }
}
```

### 4. 年次報告リマインダー（第19条対応）

cronまたはSlackリマインダーで毎年3月第3週に通知:
```
「ハローワーク求人情報提供サービス 利用状況報告期限: 4月20日」
```

## 結論

ナースロビーのハローワークAPI利用において、最も重大なリスクは**第7条(1)①イ（不特定多数への提供禁止）**の違反である。GitHubパブリックリポジトリにEXTERNAL_JOBSとしてハローワーク求人データがハードコードされている点は、即時是正が必要。

また、**第19条の年次報告期限が4月20日に迫っている**ため、これも最優先で対応すべきである。

LINE Bot内での求職者向け表示は概ね規約の趣旨に沿っているが、出典明示・労働条件の省略・独自評価の分離について改善が必要である。

社長（平島禎之）に本報告を共有し、特に年次報告の対応について即時判断を仰ぐことを推奨する。
