## 0. ビジョンと原則

### Vision
> **看護師1人が「応募する」と宣言するまで、AIが完全に伴走する。**
> **人間が触るのは "応募宣言の瞬間" 以降だけ。**

### 5つの設計原則

1. **AI-First, Human-Last**: AIで終わるなら終わらせる。人間は最終局面のみ
2. **No-Type-First**: タップ＞音声＞テキストの優先順。**ユーザーがキーボードを叩く回数を最小化**
3. **Silence-As-Default**: BOTが応答すべきか迷う時は黙る。誤発火が最大の離脱要因
4. **Conversation > Form**: 履歴書フォームに書かせない。AIが対話から抽出してシートを組む
5. **Data-Backed Replies**: AIは施設DB（84,613件）+求人DB（21,149件）+ハローワーク差分を参照して具体数字で答える

## 2. AI vs 人間の役割分担

| マイルストーン | 主担当 | 補助 | KPI |
|---------------|--------|------|-----|
| M1 友だち追加 | LP/広告 | - | フォロー率 |
| M2 希望条件 | AI（postback主体） | - | 5問完了率 |
| M3 マッチング | AI（D1検索+スコア） | - | 閲覧率/fav_add率 |
| M4 AI相談 | **AI（gpt-4o）** | - | 3ターン継続率 |
| M4.5 自己分析 | **AI（自己分析エージェント）** | - | 7問完了率 |
| M5 キャリアシート | **AI（自動生成）** | - | 生成完了率 |
| M6 応募意思 | ユーザー（明示ボタン） | - | apply_consent率 |
| **M7 応募手続き〜入職** | **人間（業務時間内）** | AI（リマインド） | 入職率 |

### 人間介入する条件（例外規則）

人間に通知が飛ぶのは**この4ケースのみ**:

1. **M6到達**: 「応募する」明示ボタンを押した瞬間
2. **EMERGENCY検出**: 8語の緊急キーワード（「死にたい」「パワハラ」等）
3. **AI 3回失敗**: 同じ質問でAIが3回ハルシネ/エラー時
4. **ユーザー明示要求**: 「人間と話したい」を明示的に押した時

それ以外は**全てAIで完結**。Slackは「監視用」であって「対応用」ではない。

## 3. AI相談の4つの起動トリガー

### Trigger 1：「気になる」直後（感情ピーク）

```
ユーザー: マッチングカードの ⭐気になる をタップ
  ↓
AI即座: 「○○病院、何が気になりましたか？
         給料 / 夜勤 / 雰囲気 / 教育体制 / その他」
        Quick Reply 5択
  ↓
ユーザー: タップ or 音声 or 自由記述
  ↓
AI: 施設DB+求人DBから具体データで回答
    例「夜勤回数は月平均8回、神奈川では平均より少なめです」
  ↓
最大12ターン会話
  ↓
「もう少し見たい」or「応募したい」or「ピンとこない」
```

### Trigger 2：マッチング後60秒沈黙（迷い）

```
ユーザー: マッチング表示 → 60秒スクロール後何もしない
  ↓
AI proactive Push:「気になることありませんか？
                    人間関係/給料/通勤時間など、
                    遠慮なく聞いてください 🎙️」
```

### Trigger 3：マッチング0件 or 「条件を変える」

```
ユーザー: 0件表示 or 「条件を変える」タップ
  ↓
AI: 「条件、もしかすると厳しめかも。
     優先順位を一緒に考えませんか？
     『絶対譲れない条件1つ』を選ぶとしたら？」
  ↓
AI診断 → 条件緩和提案 → 再マッチング
```

### Trigger 4：自由テキスト来たら（「読み取れません」廃止）

```
phase が何であれ、自由テキストが来たら
  ↓
AIに渡す（旧「読み取れません」の置換）
  ↓
AIが文脈から判断:
  ├─ 求人検索の意図 → 条件抽出 → 再マッチング
  ├─ 不安・質問 → AI相談継続
  ├─ 応募意思 → M6誘導
  └─ 不明 → 「もう少し詳しく教えてください」
```

## 4. 自己分析エージェント（M4.5の核）

### 起動条件

以下のいずれかでBOTから提案:
- M4で「応募したい」「履歴書作りたい」と発言
- AI相談3ターン以上経過
- 「気になる」を2施設以上登録

### 7問インタビュー設計

```
🎙️ 各質問は音声入力可（マイク長押しで30秒）

Q1【経験】「今まで一番長くいた職場と、何年やってきましたか？」
   → 抽出: 経験年数, 主な職場
   → 形式: 自由記述 or 音声

Q2【強み】「その職場で『これは得意』と思える業務を一言で」
   → 抽出: 専門領域, スキル
   → 形式: 自由記述 or 音声

Q3【動機】「次に転職したい理由を、率直に一言で」
   → 抽出: 転職理由（人間関係/給料/夜勤/キャリアアップ等）
   → 形式: 自由記述 or 音声 ★本音が出やすい

Q4【MUST】「次の職場で『これだけは譲れない』が1つあるとしたら？」
   → 抽出: MUST条件
   → 形式: Quick Reply 5択 + その他

Q5【NG】「逆に『これは避けたい』ものは？」
   → 抽出: NG条件
   → 形式: Quick Reply 5択 + その他

Q6【未来】「3年後、看護師としてどうありたいですか？」（任意・スキップ可）
   → 抽出: キャリア観
   → 形式: 自由記述 or 音声

Q7【補足】「最後に、担当者に伝えておきたいことはありますか？」
   → 抽出: 自由メモ（ブランク、家庭事情等）
   → 形式: 自由記述 or 音声 ★スキップ可
```

### 質問の合間で

- 共感応答（「お辛い状況ですね」「ありがとうございます」）
- 進捗バー表示（「あと3問です」）
- いつでも「あとで続ける」可能（KVに進捗保存）

### ハルシネーション対策（3段構え）

1. **禁止語リスト**: 「日本一」「絶対」「最高」等の断定表現を検出 → リトライ
2. **数字検証**: 経験年数・月給・休日数は元データと突合 → 不一致なら伏せ字
3. **ユーザー確認**: 生成後「この内容で合ってますか？」確認 → 修正リクエスト可

### KV保存先

- `member:{userId}:resume_data` (JSON、構造化データ)
- `member:{userId}:resume` (HTML、印刷用)
- `member:{userId}:career_sheet` (匿名版、病院送付用)

## 7. BOT/人間の境界制御

### 二段ガード

#### Guard 1: handoff phase 沈黙ガード（既存・強化）

```js
// worker.js:9748
if (entry.phase === "handoff") {
  // BOT沈黙、Slack転送のみ
}
```

→ 既に動いている。M6到達時に発火。

#### Guard 2: humanRepliedAt 沈黙ガード（新規追加）

```js
// !reply or chat.line.biz検知時にKVセット
entry.humanRepliedAt = Date.now()

// メッセージ受信時のチェック
if (entry.humanRepliedAt && (Date.now() - entry.humanRepliedAt) < 72*3600*1000) {
  // 72時間以内に人間が応答 → BOT全面沈黙
  // Slackに「人間対応中」と転送
}
```

### chat.line.biz検知の実装

問題: chat.line.biz経由の手動返信はworker.js経由しない

**解決策（3案）**:
- **案1（推奨）**: LINE Operator API（Standby/Active切替）でBOT/人間チャネル分離
- **案2**: 「LINE Mark As Read」イベントを受信して人間応答を推測
- **案3（運用ルール）**: chat.line.biz使用禁止、!reply or 専用管理画面のみ

→ **案1+案3併用**を推奨
