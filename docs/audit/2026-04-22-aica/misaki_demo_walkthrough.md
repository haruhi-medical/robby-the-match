# 6人パネル評定

## 1. 広告のプロ

**✅ 良い点**
- Turn 4 で「部下を守る」という具体的フレーズをそのまま拾って提案文に使うのは、広告コピー的に刺さる。本音を言葉化してくれる体験＝ブランディング
- 4ターン制限の明示は離脱を防ぐ。「最大」の単語が安心感

**⚠️ 懸念**
- Phase 9 の「進める」選択が広告着地CTAとして弱い。「応募」の重さを感じさせすぎ。「詳細を資料請求」等のソフトステップが欲しい
- Flex Carousel 1枚目カードに「CPA引き上げ要素」（看護師が求める数字: 月夜勤回数・有休取得率）が載っていない。スワイプ率低下
- LP→LINE→AICA の連携、AICAに来た時点で「何者の何のプロダクトか」の再紹介がない。初見だと戸惑う

**💰 CPA影響**
- 現状のフローは20問超の長時間セッション。1人あたり完遂率を50%として、ROAS計算に響く

## 2. UXデザイナー

**✅ 良い点**
- 1メッセージ1質問の原則が守られている
- Quick Reply選択肢が常時あり、タップで進める設計

- 4ターン後の条件カルテ表示はメンタルの区切りになって良い（達成感）

**⚠️ 懸念**
- **Phase 8→9→10→11 の連続質問が「段のないエスカレーター」化**。ミサキが「もうそろそろ終わらない？」と感じるのが Phase 10 途中

- 各phase間に「中断して続きから」の明確な案内がない
- Quick Replyの選択肢が6個並ぶ場面あり。3個くらいが理想

**🔧 具体的改善**

3. 各ターンの最後に「一旦保存して明日続ける」ボタン

## 3. 悪魔の代弁者

**🚨 致命的な穴**

1. **Phase 8 Q&Aの「1時間以内に先方確認」は嘘。実装上、社長が手動で応答する or 永遠に来ない**
   - 深夜に質問されたら1時間で応答不能
   - 嘘になった瞬間、信頼崩壊
   - → 「翌営業日までに回答します」が実態に即す

2. **書類生成の「志望動機」は AIによる創作。本人が読み直して「こんなこと言ってない」と感じる可能性**
   - 病院に誇張した志望動機が届く → 面接で齟齬
   - → 書類は**本人が語った言葉のみ**で構成する制約が必要

3. **Phase 10 の電話番号収集が早すぎる**
   - 実際の応募確定前に電話番号を渡す心理的ハードルは高い
   - 「AIなので電話来ません」の安心感と矛盾
   - → 電話番号は面接日程調整フェーズまで後回し

4. **エッジケース: 途中離脱→1週間後再開**
   - phase = JOB_QA で離脱 → 1週間後「詳しく見たい」→ presented_jobs は更新されず古い求人提示
   - → presented_jobs の TTL 実装必須

5. **軸判定の Turn 1 依存**
   - 最初の発言だけで relationship と決め、以降その軸で質問
   - ミサキの「師長が合わない + 夜勤がしんどい」は2軸だが単一軸にマッピング → Turn 2以降がズレる可能性

6. **「ロビー」ブランドなのに AIの自己紹介が「AIキャラクターの名前なし」**
   - 既存ロビーとの整合性なし
   - 覚えてもらえない

7. **音声認識の誤認識対応が甘い**
   - ack で「〇〇と聞き取りました」と表示するが、「違う、〇〇と言った」の訂正動線がない
   - 誤認識が profile_json に保存されたら取り返しがつかない

8. **Phase 12書類生成が失敗した場合の復旧が「書類お願いしますで再実行」だが、その文言を伝えるタイミングが遅い**
   - 失敗メッセージ受信後、ユーザーがブロックする確率高

## 4. 行動経済学者

**観察ポイント**

1. **授かり効果（Endowment Effect）**
   - Phase 6 の希望条件カルテ表示は強烈。一度「自分のカルテ」を持つと捨てにくくなる → 継続率UP

2. **コミットメント一貫性**
   - 4ターン心理ヒアリングで自分の問題を言語化＝コミットメント成立
   - Phase 10 で電話番号入力は一貫性原則で通過しやすいが、逆に拒否した瞬間に「自分は本気じゃない」と自己認識し離脱増

3. **損失回避**
   - Phase 9 の「応募」は損失フレーム（時間・機会）。これを「応募しないと3件の求人が無に戻る」と明示するとコンバージョン上がる
   - 例: 「この情報は7日間保存されます。7日後に消えますがお急ぎではありません」

4. **好奇心ギャップ**
   - Phase 7の Flex カードに「AI所見」（厚労省データ由来の夜勤回数・有休取得率の段階表示）があれば「これ何で分かるの」で離脱防止
   - 現状は給与・所在地など表面情報で好奇心を引けていない

5. **認知負荷**
   - 20問超の一気通貫は認知負荷過多。**3-4-5の分断構造**（心理4→条件13→個人+書類5+5）のうち、後半10問を「別日でも可」にすると完遂率上昇

6. **社会的証明**
   - 「同じ6年目の先輩も、この病院を選んでいます」的な文言は現状ゼロ
   - （ただしデータがないので作れない、嘘は避けるべき）

## 5. LINE Bot開発者

**✅ 技術的に良い点**
- reply token の消費管理（音声対応で null→pushフォールバック）が綺麗
- JSON Modeでのextracted merge + is_completeの判定が堅い
- 3段フォールバック（リトライ→不足項目→静的要約）の防御深い

**⚠️ 技術的懸念**
1. **AICA の Worker と既存 Worker の `LINE_SESSIONS` KV が分離**
   - 音声メッセージの messageId は24時間でLINE側から消える。24時間以内に取得必須
   - 音声ファイルが25MB超の場合、LINE側仕様で取得は可能だがWhisperに渡せない

2. **D1 の `profile_json` が TEXT カラム。サイズ制限2MBに近づくと select が重い**
   - 特に interview_mock_qa 履歴 + presented_jobs + その他で肥大化リスク

3. **LINE Reply Token は10分。書類生成に30-40秒かかる場合、生成中のユーザー発言への reply が失敗する可能性**
   - 現状は「生成中です、お待ちください」reply を先に返すので大丈夫
   - ただし Whisper transcription が遅延すると reply token が先に切れる

4. **cron triggers（`0 0 * * *` と `0 * * * *`）は設定済みだが scheduled() ハンドラが no-op**
   - follow_ups テーブル活用が未実装
   - 入職後フォロー Push の土台が未整備

5. **/health エンドポイントに各依存（OpenAI, D1, KV）の疎通チェックがない**
   - 障害の早期検知ができない

6. **Worker CPU時間制限**
   - Whisper + OpenAI 書類生成3件並列 + D1更新 の合算が30秒超えたら中断リスク
   - 実測確認必要

# 全員一致の結論

**AICA は方向性として正しい。しかし完遂率が最大の課題。**

- Phase 1-8（心理→条件→マッチング→Q&A）は通過率高
- Phase 9-11（応募確定→個人情報→書類ヒアリング）が最大離脱ゾーン
- Phase 12-13（書類生成→レビュー）は高価値だがPhase 9-11を越えないと到達しない
