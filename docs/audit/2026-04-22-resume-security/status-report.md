## 3. 発見した脆弱性（監査結果）

### 🔴 Critical（即時対応済）

#### 3-1. 閲覧URLに認証が一切なかった
- 旧URL例: `https://.../api/resume-view/1ef7256b-818`（12文字のランダム文字列）
- **URL を知る者は誰でも閲覧可能**な設計
- ID長12文字（約44bit相当）、理論上ブルートフォースは困難だがベストプラクティスから外れる

#### 3-2. 履歴書生成APIに認証・使用回数制限なし
- `/api/resume-generate` エンドポイントは認証ゼロ
- 任意のLINE user_id を指定して「履歴書が完成しました」という**偽装LINEメッセージを第三者に送信可能**（フィッシング悪用の温床）
- OpenAI APIコストを爆発的に消費させる攻撃（1リクエスト約$0.002）も可能

#### 3-3. 個人情報取扱同意画面の欠落
- OpenAI社（米国）への越境データ移転について、個人情報保護法第28条が要求する**本人の明示的事前同意を取得していなかった**
- 違反時は個人情報保護委員会の指導・勧告、最大罰金1億円（法人）の対象

### 🟡 High / Medium（即時対応済）

#### 3-4. 入力バリデーションの欠如
住所・職歴・自由記述に文字数制限なし → 大容量データの送り付けによるコスト・容量攻撃の余地

#### 3-5. LINE userId がURLクエリ露出
`resume/?user_id=XXX` の形でブラウザ履歴・スクショにuser_idが記録されていた

#### 3-6. クライアント送信値の無検証
`sessionId` `userId` をクライアントが送った値のまま信用

## 4. 本日実施した対策（2026-04-22）

### 4-1. 認証・レート制限の導入
- **短期トークン方式**: LINE Bot が「履歴書を作成する」押下時に30分有効のトークンをサーバー発行 → Webフォーム送信時に検証 → 使い切り削除
- **IPベース利用制限**: 同一IPから5回/24時間の上限
- **userIdはサーバー側で復元**: クライアント送信値は信用しない

### 4-2. 閲覧URLの強化
- ID長を12文字 → **36桁UUID（2^128通り）** に拡張、ブルートフォース事実上不可能に
- レスポンスに以下のセキュリティヘッダー付与:
  - `Referrer-Policy: no-referrer`（外部サイトへのURL漏洩防止）
  - `Cache-Control: no-store`（ブラウザキャッシュ残存防止）
  - `X-Frame-Options: DENY`（iframe埋め込み攻撃防止）
  - `X-Content-Type-Options: nosniff`（MIMEスニッフィング防止）
  - `X-Robots-Tag: noindex, nofollow`（検索エンジン除外）
- **既存の7件URL（昨日の実ユーザー分含む）は引き続き閲覧可能**（移行期間として正規表現を旧新両対応に）

### 4-3. 個人情報同意の取得
- フォームに必須チェックボックス2つを追加:
  1. ☐ 個人情報保護方針への同意＋履歴書作成業務への利用承諾
  2. ☐ OpenAI社（米国）への情報送信への同意
- 2つとも未チェックで送信ボタン無効＋サーバー側でも検証

### 4-4. プライバシーポリシー改定
`privacy.html` に第11条（AI履歴書作成機能における個人情報の取り扱い）を新設:
- 取得する情報項目の明示
- OpenAI社への越境移転の明示と保持期間（最大30日）
- 生成履歴書の保存期間（7日）と自動削除
- 閲覧URLが推測不能な36桁識別子であること
- 削除請求窓口の明示

### 4-5. 検索エンジン対策の二重化
`robots.txt` に `Disallow: /resume/` を追加（既存の `noindex` メタタグと合わせて二重防御）

### 4-6. 入力バリデーション
住所200字、職歴30件、自由記述1,000〜2,000字等の上限を設定

## 6. 残リスクと今後の選択肢

### 6-1. 残存リスク: 「URLを入手した者は認証なしで見れる」設計

| 漏洩経路 | 現在の対策 | 残リスク |
|---|---|---|
| Googleインデックス | noindex + robots.txt + sitemap未掲載 | ほぼゼロ |
| Refererでの外部漏洩 | Referrer-Policy: no-referrer | ゼロ |
| ブルートフォース推測 | 36桁UUID（2^128） | ゼロ |
| **本人がSNS等にURLをコピペ** | なし | **残** |
| **本人のLINE/端末が乗っ取られる** | なし | **残** |
| **Slack/LINEのトークン漏洩** | 各社のアクセス制御に依存 | 残 |

担当者が閲覧できるよう認証を設けていないため、「URLを持つ者=閲覧権限者」という設計である点は変わらない。

### 6-2. 完全に塞ぐ3つの選択肢（要判断）

#### A案. HMAC署名付き短期URL（推奨・工数2時間）
- URL構造: `/api/resume-view/:id?sig=xxx&exp=yyy`
- サーバー側で HMAC-SHA256 署名と有効期限を検証
- 有効期限例: 24時間（本人が印刷/PDF保存するのに十分、漏洩時の被害窓を24hに限定）
- 期限切れ後は LINE Bot のリッチメニュー「履歴書をもう一度見る」で新URLを再発行
- ユーザー体験は実質変わらず

#### B案. LIFF+LINE userId 本人認証（工数1日）
- 本人アクセス: LIFF SDK でLINEのuserId取得 → KVの紐付けと照合したときのみHTML表示
- 担当者アクセス: 別URL（HMAC署名付き）
- 利点: 本人以外は URL をSNSにコピペしても他人には開けない
- 欠点: 担当者運用の別窓口が必要、UXに認証画面が挟まる

#### C案. 1回使い切り（工数1時間）
- 初回アクセスでHTML返却→即KV削除
- 欠点: 印刷前にリロードすると消える、担当者との共有不可 → **実質運用不可**

#### 現状維持（対応なし）
- 36桁UUID + 4-2セキュリティヘッダーで実用上十分と判断する立場
- Googleインデックスや推測による漏洩はゼロ、残リスクは本人由来のみ

**推奨**: A案を本日中に追加実装。2時間でユーザー体験を損なわずに漏洩被害窓を24hに限定できる。

## 7. 法的整理（個人情報保護法・職業安定法）

### 7-1. 職業安定法
はるひメディカルサービスは**有料職業紹介事業者**として許可を受けており、求職者からの個人情報取得は事業の本旨として当然合法（職安法第5条の4 + 厚労省指針）。

### 7-2. 個人情報保護法

| データの取得者 | 法的分類 | 必要な対応 | 現状 |
|---|---|---|---|
| 自社キャリアアドバイザー（社長・担当者） | **内部利用**（第三者提供に非該当） | 利用目的の通知のみ | ✅ privacy.html 第4条で明示済み |
| OpenAI（米国） | **越境第三者提供**（法28条） | 明示的事前同意必須 | ✅ 本日フォームに同意チェックボックス追加 |
| 求人医療機関（病院） | **国内第三者提供**（法27条） | 送付時に都度本人同意 | ✅ 既存フローで対応 |

### 7-3. 「自社で見る」ことの整理（代表からの質問への回答）
- 「第三者提供」は**別法人**に渡すことを指す。自社従業員・代表が閲覧するのは第三者提供に該当しない。
- 利用目的の範囲内（履歴書作成・求人紹介業務）で閲覧する限り、本人の追加同意は不要（法18条）。
- 必要なのは:
  1. 利用目的の事前通知 → ✅ privacy.html + フォーム同意で対応済み
  2. 安全管理措置 → ✅ privacy.html 第6条で明示
  3. 従業員への監督 → ✅ 雇用契約内の守秘義務

### 7-4. 運用上の注意点（ステークホルダーに共有すべき事項）

1. **運営主体の統一性**
   - 「ナースロビー」は屋号/サービス名、法人格は「はるひメディカルサービス」1つという建付け。この建付けを崩すと第三者提供扱いとなり同意取得フローが変わる

2. **業務委託者の扱い**
   - 今後アドバイザーを業務委託（フリーランス等）で雇う場合、個人情報保護法第25条の「委託先」扱いとなり**監督義務が生じる**
   - 対応: 業務委託契約に守秘義務条項・個人情報取扱条項を必ず入れる

3. **Slack/LINE経由のアクセス**
   - Slack・LINEは米国企業のサービスのため、業務委託先（法25条）扱い
   - Slackのリンクプレビュー機能で履歴書HTMLのプレビュー画像がSlackサーバーに生成・キャッシュされる可能性あり
   - 対策: Slack通知本文のURLを `<URL>` 記法で送信するとプレビュー無効化可能（5分で実装可能）

4. **病院への履歴書送付は別手続き**
   - 第三者提供に該当するため、本人に「この病院に送付します」と個別同意を都度取る
   - 既存の紹介フローで対応済みだが、運用徹底が必要
