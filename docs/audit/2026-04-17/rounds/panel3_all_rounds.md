# Round 1: 6人独立点検

## 専門家1: D1データベース専門家

### 発見した阻害要因

1. **全体カバー率の誤認識（100% vs 6.1%）**
   - 根拠: FACT_PACK「診療科100%」は病院1,498件のサブセット。ローカルSQLiteでクリニック22,978件の departments=0件（`SELECT COUNT(*) FROM facilities WHERE departments!='' AND category='クリニック'` = 0）。DB全体では **1,497/24,488=6.1%**
   - 改善案: マッチングで診療科フィルタを使う場合、病院カテゴリに限定する旨をUIで明示。クリニック検索時は診療科条件を適用しない（データなしで0件になる）
   - インパクト: クリニック希望者の診断「0件」問題を解消（ミサキが診療科「内科」を選ぶとクリニック22,978件から0件になる）
   - 実装コスト: 2h（L4890 `departments LIKE` のガード追加）
   - 依存: なし

2. **訪問看護ST 0件問題**
   - 根拠: `SELECT DISTINCT category FROM facilities` → 病院/クリニック/介護施設のみ。訪問看護ST=0件。`CATEGORY_MAP.visiting='訪問看護ST'`（worker.js L619）が常時不一致
   - 改善案: 訪問看護求人はD1 `jobs` テーブル側でemployer/title LIKE '%訪問看護%' で検索。facilitiesフォールバックを無効化
   - インパクト: 訪問看護希望者の「0件→handoff」誘導を削減。ミサキ「訪問看護興味ある」の離脱防止
   - 実装コスト: 3h（L4870 D1施設フォールバックのbypass追加）
   - 依存: STATE.md「訪問看護STデータ投入 8-12h」が予定されているが未実施

3. **介護施設12件＝実質ゼロ**
   - 根拠: 介護施設12件はすべて東京都（神奈川0/千葉0/埼玉0）。県西・湘南等の介護求人希望者への提案不可
   - 改善案: ハローワークjobsから `employer LIKE '%老人%' OR '%介護%' OR '%特養%'` で抽出し facilities に逆補完（再構築不要）
   - インパクト: 介護希望者の適合求人0件問題の緩和
   - 実装コスト: 4h

4. **has_active_jobs=0件 = 施設↔求人リンク機能停止**
   - 根拠: `SELECT COUNT(*) FROM facilities WHERE has_active_jobs=1` = 0
   - 改善案: hellowork_to_d1.py で施設名×住所で facilities.id ルックアップ → jobs.facility_id 更新 + facilities.has_active_jobs=1
   - インパクト: 「求人あり施設」フィルタUIが動作する。成約確度の高い施設だけ提示可能
   - 実装コスト: 6h
   - 依存: なし

5. **データ鮮度: facilities 2026-04-03固定（14日前）**
   - 根拠: `SELECT MAX(last_synced_at) FROM facilities` = 2026-04-03。厚労省公的データ更新サイクル年1回なので問題小だが、明示なし
   - 改善案: 施設DBは半年に1回更新の方針をSTATE.mdに明記（STATE.md「DB自動更新cron…やらないリスト」と整合）
   - インパクト: 小。ドキュメント整合性のみ
   - 実装コスト: 0.5h

## 専門家2: マッチングロジック専門家

### 発見した阻害要因

1. **scoreFacilities が facilities = 212施設の旧DB（config.js）依存**
   - 根拠: worker.js L885-L897 `scoreFacilities` の入力は `FACILITY_DATABASE`（config.js 212施設）。D1の24,488施設はLINE Botのマッチングでしか使われない。LP側の診断（chat.js）では**古い212施設**がソースのまま
   - 改善案: chat.js 用の推論エンドポイント（LP Worker）を D1 facilities にフォールバック。または LP診断は D1 `jobs` の高スコア順表示に切り替え
   - インパクト: LP流入→LP診断→LINE遷移の品質が揃う。重大な「リソース分離」問題
   - 実装コスト: 8h
   - 依存: LP v2 検討タスクと一致

2. **逆マッチ方式（病院=訪問/クリニック/介護でないもの）の脆弱性**
   - 根拠: L4823 `matchFlags.facilityType = !isVisiting && !isClinic && !isCare`。hellowork求人のタイトルが「ケアプラザ」「老健」「デイサービス」のキーワード漏れがあると病院扱いされる。L280（川崎）のジール・チャイルドケア（保育園）→ 「病院」マッチで残存
   - 改善案: ポジティブマッチ `isHospital = /病院|病棟|総合病院|医療センター/.test(text)` を導入。ネガティブ判定を補完
   - インパクト: 病院希望者の不適合求人混入を削減（保育園61件・介護類の誤混入）
   - 実装コスト: 2h

3. **ADJACENT_AREAS が越境を過剰に許容**
   - 根拠: L4610 `tokyo_included: ['yokohama_kawasaki']`、L4603 `yokohama_kawasaki: [..., 'tokyo_included']`。横浜ミサキが0件→東京求人に自動拡大→通勤できない求人を提示
   - 改善案: 通勤時間による閾値 or ユーザー同意タップを挟む（「隣接エリアも見ますか？」）
   - インパクト: 誤マッチの離脱防止。handoff時の電話で「ここ通えない」と言われるリスク削減
   - 実装コスト: 3h

4. **D1 jobs検索の ORDER BY score DESC で「Dランク求人」も候補**
   - 根拠: L4725 `ORDER BY score DESC LIMIT 15`。Dランク906件（全体30.6%）がscore昇順でも混入する可能性。除外フィルタなし
   - 改善案: `WHERE rank IN ('S','A','B')` を追加。3件未満ならCを許容
   - インパクト: 低品質求人を紹介候補から除外。成約確率↑
   - 実装コスト: 1h

5. **同一事業所dedup後5件まで」で選択肢が偏る**
   - 根拠: L4750 `employerCount[emp] <= 2`、dedup後 `.slice(0, 5)`。神奈川エリアでスコア上位に同一法人（例：晃友会3件）が固まると、多様性が消える
   - 改善案: dedup閾値を1に厳格化。施設カテゴリ別に最低1件ずつ含める
   - インパクト: 求人多様性↑、ミサキの「選んだ気」促進
   - 実装コスト: 2h

6. **departmentsフィルタで D1フォールバックが常時0件**
   - 根拠: L4890 `departments LIKE ?`。クリニック/介護22,990件に departmentsが入っていない → 診療科指定時は病院1,498件しかヒットしない
   - 改善案: クリニック検索時は departmentsフィルタを無効化。あるいは `departments IS NULL OR departments LIKE ?`
   - インパクト: 「内科希望×クリニック」パターンのマッチ実現
   - 実装コスト: 1h

横浜ミサキ「急性期・内科」でマッチ → Dランク混入・ジール（保育園）混入・晃友会偏り → Yes 止まる

## 専門家3: AI応答品質専門家

### 発見した阻害要因

1. **4段フォールバックは存在しない（if/else if/else の1段分岐）**
   - 根拠: worker.js L1775-L1851。`AI_PROVIDER=openai` のときOpenAI失敗→502応答（L1800）、Claude/Geminiへのフォールバックなし。ドキュメント（STATE.md, FACT_PACK）と実装が乖離

   - インパクト: OpenAI障害時の会話継続率↑、離脱防止
   - 実装コスト: 3h
   - 依存: ANTHROPIC_API_KEYがSTATEで「未設定」。Workers AI（Llama 3.3 70B）は設定あり→少なくとも2段リトライ可能

2. **AI応答ターン上限5での打ち切りがハンドオフ誘導と対立**
   - 根拠: L1861 `done = userMsgCount >= 5`。STATE.md「AI相談ターン上限5設定」。ミサキが6問目を入れた瞬間終了→誘導不足
   - 改善案: 5ターン到達時に「担当者に引き継ぎますか？」のQR提示（強制切断ではなく選択肢）
   - インパクト: 高エンゲージメントユーザーの取りこぼし防止
   - 実装コスト: 1h

3. **緊急キーワード検出の粒度が粗い**
   - 根拠: L6822 `EMERGENCY_KEYWORDS = ['死にたい','自殺','限界','もう無理','パワハラ',...]`。「限界」は日常語でも出る（「今月シフトが限界」→handoff誤発動）

   - インパクト: 誤handoffによるBot沈黙・ミサキ戸惑い解消
   - 実装コスト: 0.5h

4. **OpenAI認証切れ監視なし**
   - 根拠: wrangler secret で設定済みだが、shutown 後に再設定しないとすべてのLINE応答が502
   - 改善案: watchdog.py に「LINE Bot /api/chat でOpenAI 401/403をモニタ」追加。5分以内にSlack通知
   - インパクト: SLO確保、ユーザー無応答事故を抑制
   - 実装コスト: 2h

5. **Flex Messageフォールバック（空テキスト）の既知バグ対策は実装済みだが検証なし**
   - 根拠: MEMORY「Flex Message空テキスト→フォールバック値必須」。実装箇所は buildFacilityFlexBubble のスライス処理（L4961-）で対応。ただし全分岐テストは未実施
   - 改善案: unit test追加（1件のテスト）
   - インパクト: 小
   - 実装コスト: 1h
