# ナースロビー 2段階メンバーシップ制 設計書

- 設計日: 2026-04-22
- 対象: ナースロビー（quads-nurse.com）全体の会員制リニューアル
- 運営: はるひメディカルサービス
- ブランチ: main
- ステータス: 設計承認済 → 実装計画作成待ち

---

## 1. 背景と解決する矛盾

現状のナースロビーは、**LINE友だち追加を事実上の「会員登録」として運用**している。ウェルカムメッセージ・AIチャット・求人マッチング・履歴書作成など、LINE追加だけで全機能にアクセスできる設計。

一方で履歴書作成（2026-04-21 から稼働）では、別途フォームで氏名・電話・住所・学歴・職歴を入力させる。ユーザー体験としては**二重登録**の違和感を生じており、代表から「LINE追加を会員登録とみなしているのに、履歴書でまた登録するのは矛盾」との指摘があった。

**本設計はこの矛盾を「2段階メンバーシップ制」で解消する**。LINE追加=ビジター（軽い接触）、履歴書作成 or お気に入り登録=ナースロビー会員（継続的な関係）。履歴書作成フォームの同意チェックを「会員登録の明示的な意思確認」として機能させ、詐欺感のない自然な会員化を実現する。

あわせて、2026-04-22 に発見した「履歴書閲覧URLに認証がない」セキュリティ設計上の残リスクを、LIFF認証ベースのマイページ化で根本解消する。

## 2. ビジョン

> LINE で静かに、転職活動。履歴書を作った瞬間からナースロビーに自分の席ができる。

- ビジターでも十分に役立つサービスを維持（求人検索・AI相談は従来通り無料）
- 「履歴書を作る」という最も能動的な行動を会員化のトリガーとする
- 会員には「保存される」「AIが最高のものをくれる」「継続的な関係」を提供
- ブランド「静かに転職活動」を一切損なわない。通知は最小限、マイページは静かな個人アーカイブ

## 3. ユーザー区分

### 3-1. ビジター（LINE友だち追加のみ）
| 項目 | 内容 |
|---|---|
| トリガー | LINE公式アカウント友だち追加 |
| 保存データ | LINE user_id と会話セッション（24h） |
| できること | LINE Botで都度「求人見せて」「相談」/ AIチャット / マッチング検索 |
| できないこと | 履歴書作成/保管、お気に入り、条件保存、新着LINE配信 |
| ブランド訴求 | 「名前も聞きません。LINEで静かに」 |

### 3-2. ナースロビー会員
| 項目 | 内容 |
|---|---|
| トリガー（MVP-A） | 履歴書作成フォームの同意チェック2つ + 送信 |
| トリガー（Phase 2） | お気に入り求人保存時の最小プロフィール入力 + 同意 |
| 保存データ | 個人情報（氏名・電話・住所・職歴等）、希望条件、お気に入り求人 |
| できること | ビジターの全機能 + 履歴書保管/再編集/PDF、お気に入り、条件保存、AI新着LINE配信 |
| ブランド訴求 | 「あなた専用の席ができる」「AIが最高のものをくれる」 |

## 4. 会員価値の4本柱

|---|---|---|---|

## 5. 会員化フロー（ルートA / MVP-A対象）

```
[LINE Bot]
   │  ユーザーがリッチメニュー「履歴書を作成する」タップ
   ▼
[LINE Bot → Worker] rm_resume_start フェーズ
   │  短期トークン発行（30分有効、KV: resume_token:<uuid>）
   │  「履歴書を作成しましょう」+ URL `https://quads-nurse.com/resume/?token=XXX`
   ▼
[ブラウザ（quads-nurse.com/resume/）]
   │  LIFF起動で LINE user_id 照合（任意、トークンで代替可）
   │  フォーム表示・入力
   │  最下部の同意チェック2つ:
   │    ☐ 利用規約・個人情報保護方針に同意し、ナースロビー会員として登録します（必須）
   │    ☐ OpenAI社（米国）で処理することに同意します（必須）
   │  送信
   ▼
[Worker] POST /api/resume-generate
   │  トークン検証 → KVから user_id 復元
   │  入力バリデーション・同意チェック
   │  AI志望動機生成（OpenAI GPT-4o-mini）
   │  履歴書HTML生成
   │  KV保存: member:<userId> + member:<userId>:resume + member:<userId>:resume_data
   │  従来の resume:<id> への同時保存は廃止（会員時代は member配下に一本化）
   │  トークン削除（1回使い切り）
   │  Slack通知 / LINE Push（既存）
   ▼
[ブラウザ]
   │  完成画面にリダイレクト「✨ ナースロビー会員になりました」
   │  「📄 履歴書を見る・印刷する」ボタン → PDF表示
   │  「🏠 マイページへ」ボタン → /mypage/
   ▼
[LINE Bot]
   │  別途Pushメッセージ「マイページはこちら → quads-nurse.com/mypage/」
   │  リッチメニュー自動切替（会員用メニューへ）
```

### 5-1. ルートB（Phase 2 で追加・本設計書では骨子のみ）
- LINE Botのお気に入りボタン押下時に「お気に入りを保存するには会員登録が必要」のメッセージ
- 最小プロフィールフォーム（氏名・電話・希望エリアのみ）で会員化
- 履歴書は後でゆっくり作れる

## 6. 認証基盤

### 6-1. LIFF（LINE Front-end Framework）
既存の LIFF ID `2009683996-7pCYfOP7` を流用。マイページアクセス時に LIFF SDK を初期化し、LINE user_id を取得してサーバー側 KV レコードと照合。本人のみが自分のマイページを開ける設計。

### 6-2. 認証フロー
```
[ユーザー] mypage/ にアクセス
   ▼
[ブラウザ] LIFF SDK初期化 → liff.isLoggedIn() 判定
   │
   ├─ LINEブラウザ内 & ログイン済 → user_id 即取得 → 次へ
   │
   └─ 外部ブラウザ等 → 「LINEから開いてください」の案内画面表示（リッチメニューへ誘導）
   ▼
[ブラウザ] POST /api/mypage-init { userId }
   ▼
[Worker] KVで member:<userId> を検索
   ├─ あり → 会員情報とセッショントークン（HMAC署名付き、24h有効）を返す
   └─ なし → 404 「まだ会員登録されていません」+ 履歴書作成への誘導
   ▼
[ブラウザ] セッショントークンを localStorage に保存（24h）
   マイページUIレンダリング
   各API呼び出し時にセッショントークンをAuthorizationヘッダーに付与
```

### 6-3. 担当者（キャリアアドバイザー）のアクセス
- 担当者は LIFF 認証を経由せず、既存の Slack通知に含まれる URL から直接閲覧
- Slack通知URLは **HMAC署名付き短期URL（24h有効）** に変更（2026-04-22 の残タスク #1 A案）
- 期限切れ後は LINE Bot 側で再発行可能（担当者用コマンド）

## 7. データモデル (Cloudflare KV)

### 7-1. 会員プロファイル
```
Key:   member:<userId>
Value: {
  userId: "U123...",
  createdAt: 1234567890000,
  consentedAt: 1234567890000,
  displayName: "田中　美咲",    // 履歴書データから取得、匿名化表示可
  status: "active",            // "active" | "deleted" | "suspended"
  version: 1
}
TTL: なし（会員は永続、退会時のみ削除）
```

### 7-2. 会員の履歴書HTML（表示用）
```
Key:   member:<userId>:resume
Value: 履歴書HTML（プレースホルダー置換済み）
TTL:   なし（会員継続中は保持、退会/削除時にクリア）
```

### 7-3. 会員の履歴書元データ（再編集用）
```
Key:   member:<userId>:resume_data
Value: JSON（フォーム送信時の data そのまま。再編集時のフォーム初期化用）
TTL:   なし
```

### 7-4. 既存のキー（維持）
```
resume_token:<token>   既存の30分トークン、変更なし
session:<sessionId>    既存のLINE会話セッション、変更なし
resume:<id>            既存の12文字ID／36文字UUID、deprecateするが削除はしない（4/27の自然失効待ち）
```

### 7-5. 会員マイページ用セッショントークン
```
Key:   mypage_session:<userId>:<tokenId>
Value: { issuedAt, expiresAt, ipHash }
TTL:   24時間
```

## 9. API設計（MVP-A）

### 9-1. 新規エンドポイント
```

  Body: { userId }
  Response: { memberFound: true/false, sessionToken, displayName, resumeUpdatedAt }
  認証: なし（LIFFからの初回照合、レート制限でBot回避）

  Header: Authorization: Bearer <sessionToken>
  Response: 履歴書HTML（member:<userId>:resume）
```

## 11. セキュリティ設計

### 11-1. 2026-04-22 の残リスク解消状況
| リスク | 解消方法 |
|---|---|
| 閲覧URLを持つ人誰でも見れる | 本人: LIFF認証+セッショントークン / 担当者: HMAC署名付き短期URL |
| URLをコピペでSNS共有 | LIFF認証なしでは開けない |
| 担当者の端末が盗まれる | HMAC URLの24h失効 |
| Slackトークン漏洩 | HMAC URLの24h失効+URLが短期で無効化 |

### 11-2. 既存のセキュリティ継承
- `X-Robots-Tag: noindex, nofollow`
- `Referrer-Policy: no-referrer`
- `Cache-Control: no-store`
- 入力バリデーション（字数上限）
- IPレート制限（5/24h）
- 同意チェック2つ必須

### 11-3. 新規追加
- マイページ API すべてにセッショントークン必須
- セッショントークンは HMAC-SHA256 署名 + 24h 有効
- LIFF の ID token 検証（本人確認の強化、Phase 2 で導入可）
