# 施設検索機能 — 25視点の設計分析

> 作成: 2026-04-06 | D1 facilities 24,488件 / D1 jobs 2,936件

---

## A. 何を検索するのか（10視点）

### 1. 施設名で検索（「〇〇病院」「△△クリニック」）

**現状データ**: `facilities.name` に正式名称が入っている。法人名付きの長い名称（例: `社会福祉法人恩賜財団済生会支部東京都済生会 東京都済生会中央病院`）が多い。

**問題点**:
- ユーザーは「済生会中央病院」と入力するが、正式名称は60文字超。部分一致（LIKE `%済生会中央%`）が必須
- 同名施設の存在（例: 「中央病院」で複数ヒット）→ 地域で絞り込む必要
- クリニック22,978件の多くは「○○クリニック」で似た名前が大量にある

**設計方針**: `LIKE '%keyword%'` で十分。FTSは不要（後述 #21）。入力補完（オートコンプリート）は初期実装では見送り、テキスト入力→結果一覧の方がシンプル。

**優先度**: ★★☆（ニッチ。特定施設を調べたい人向け。メイン導線にはならない）

---

### 2. エリアで検索（「横浜市」「柏市」）→ 既存の求人検索との違いは？

**既存**: LINE Botのintake_lightフロー（Q1-Q3）でエリア選択 → マッチング結果表示。これは「求人がある施設」のみ。

**施設検索との違い**:
| | 求人検索（既存） | 施設検索（新規） |
|---|---|---|
| 対象 | jobs 2,936件（ハロワ求人） | facilities 24,488件（全施設） |
| 表示条件 | 募集あり | 募集なくても表示 |
| ユーザーの意図 | 「今すぐ転職したい」 | 「この病院ってどんなところ？」 |
| コンバージョン | LINE登録→面談 | 情報収集→将来のLINE登録 |

**核心的な問い**: 求人がない施設を見せて何のメリットがあるのか？

**答え**: 
- 「気になる施設について聞いてみる」→ handoff → 逆指名リクエスト。これが事業の核心（病院はオーダーメイドで求人を出す）
- SEO的価値: 「○○病院 看護師 口コミ」で流入 → LINE登録
- 24,488件 vs 2,936件。22倍のコンテンツ量で検索エンジンに対して圧倒的に有利

**設計方針**: エリア検索はAREA_CITY_MAPをそのまま流用。`address LIKE '%横浜市%'` or `prefecture = '神奈川県'`。既にworker.jsに実装済みのロジックを流用可能。

**優先度**: ★★★（メイン導線の1つ）

---

### 3. 最寄駅で検索（「横浜駅から近い病院」）

**現状データ**: `nearest_station`（例: `東京メトロ千代田線 大手町`）、`station_minutes`（徒歩分数）。

**問題点**:
- `nearest_station`は路線名+駅名のフォーマット。「横浜駅」で検索するなら `LIKE '%横浜%'` だが、「横浜」を含む駅は横浜、新横浜、東横浜など多数
- 徒歩分数でソートできるのは強み（「駅チカ」ニーズ）
- クリニック22,978件のうち`nearest_station`がNULLの割合は未確認。データ品質次第

**設計方針**: `nearest_station LIKE '%駅名%' ORDER BY station_minutes ASC`。路線名は無視してOK。

**優先度**: ★★☆（ミサキは通勤時間を重視するが、駅名入力より「エリア+駅チカ」フィルタの方が自然）

---

### 4. 診療科で検索（「循環器科がある病院」）

**現状データ**: `departments` カンマ区切り（例: `内科,外科,循環器内科,呼吸器内科...`）。

**強み**: 24,488件中、病院1,498件はほぼ全件に診療科データあり。クリニックも大半あり。

**ユースケース**: 「オペ室経験を活かしたい → 外科のある病院」「NICU経験 → 小児科・産婦人科のある病院」

**問題点**:
- 診療科名のバリエーション（「循環器科」vs「循環器内科」vs「心臓血管外科」）。ユーザー入力をどう正規化するか
- `LIKE '%循環器%'` で広めに拾うのが現実的

**設計方針**: 主要診療科をボタンで選ばせる（自由入力ではなく）。内科/外科/整形外科/循環器内科/小児科/産婦人科/精神科/リハビリテーション科/眼科/皮膚科あたりの10-15個。

**優先度**: ★★★（看護師は「何科で働くか」が転職の最重要軸の一つ）

---

### 5. 条件で検索（「託児所あり」「夜勤なし」）→ facilitiesにこのデータはない

**現実**: D1 facilitiesには福利厚生・勤務条件のカラムがない。ハロワ求人(jobs)には一部あるが、施設マスターには紐付いていない。

**取得可能性**:
- 託児所あり: 厚労省データにはない。各施設のWebサイトから個別にスクレイピングが必要 → 膨大な工数
- 夜勤なし: これは施設の属性ではなく求人の属性。jobsテーブルの`shift_type`に「日勤のみ」がある
- 電子カルテ: 未取得
- 寮あり: 未取得

**設計方針**: 初期実装では条件検索は**実装しない**。データがないのに検索UIだけ作っても失望体験になる。将来的にjobsテーブルから「夜勤なし求人あり」フラグを施設に逆反映する程度。

**優先度**: ★☆☆（データがないので実装不可。優先度を上げるなら先にデータ収集が必要）

---

### 6. 規模で検索（「300床以上の大病院」「小規模クリニック」）

**現状データ**: `bed_count`（病床数）。病院1,498件はほぼ全件にデータあり。クリニックはNULL or 0が多い（入院設備なし）。`nurse_fulltime`（常勤看護師数）も病院はかなり埋まっている。

**ユースケース**:
- 「大学病院クラスでバリバリ働きたい」→ 500床以上 + DPC特定病院群
- 「アットホームな環境がいい」→ 100床以下 or クリニック
- 「看護師が多い=教育体制が整っている」→ nurse_fulltime 200名以上

**設計方針**: レンジスライダーではなくプリセットボタン: 「大規模（300床+）」「中規模（100-299床）」「小規模（100床未満）」「クリニック」。`dpc_group`（DPC特定病院群/標準病院群/非DPC）も判別材料になる。

**優先度**: ★★☆（ミサキにとって「病院の雰囲気」は重要だが、病床数はその間接指標でしかない）

---

### 7. 「自宅から近い」で検索（位置情報）

**現状データ**: `lat`, `lng` が全施設に入っている。`STATION_COORDINATES` もworker.jsに定義済み。

**技術的に可能**: ユーザーの位置情報（GPS or 住所入力）→ ハバーサイン距離計算 → 近い順にソート。D1のSQLiteでは `(lat - ?)^2 + (lng - ?)^2` の近似計算で十分（関東圏なら誤差無視可能）。

**問題点**:
- LINE Bot上で位置情報を取得する方法: LINEのLocation Messageを使えば可能（「位置情報を送信してください」）
- Webの場合: Geolocation API → HTTPS必須（対応済み）
- 24,488件の距離計算: インデックスなしだとD1で重い → `lat`/`lng`の範囲指定で先に絞る（BBox filter）

**設計方針**: 
```sql
-- Step 1: BBox filter（±0.1度 ≈ ±11km）
SELECT * FROM facilities 
WHERE lat BETWEEN ? - 0.1 AND ? + 0.1 
AND lng BETWEEN ? - 0.1 AND ? + 0.1
-- Step 2: アプリ側で距離計算してソート
```
`idx_facilities_lat_lng` インデックスは既に存在する。

**優先度**: ★★★（「通勤圏内」は看護師の転職における最重要条件。LINE位置情報送信と組み合わせれば非常に自然なUX）

---

### 8. 複合検索（「横浜市の循環器科がある200床以上の病院」）

**技術的**: WHERE句をANDで繋ぐだけ。D1のSQLiteで問題なし。

```sql
SELECT * FROM facilities 
WHERE address LIKE '%横浜市%' 
AND departments LIKE '%循環器%' 
AND bed_count >= 200 
AND category = '病院'
ORDER BY bed_count DESC
LIMIT 20
```

**問題点**:
- 条件を増やすほど結果が0件になるリスク → 「条件を緩めてみませんか？」の提案が必要
- UI設計: 全条件を一度に入力するフォームは複雑すぎる → 段階的絞り込み（エリア→診療科→規模）の方がミサキに合う

**設計方針**: LINEのFlex Messageで段階的に条件追加。「もっと絞り込む？」→ ボタンで条件選択。

**優先度**: ★★☆（基盤として必要だがUIは段階的絞り込みで対応）

---

### 9. ユーザーは何を期待して「施設検索」をタップするのか？

**ミサキ（28歳看護師）のシナリオ**:
1. 「今の職場の近くに他にどんな病院があるか見たい」→ エリア検索
2. 「友達が○○病院に転職したって言ってた。どんな病院？」→ 施設名検索
3. 「循環器に強い病院で働きたい」→ 診療科検索
4. 「大きい病院で研修したい」→ 規模検索
5. 「とりあえず求人あるところだけ見たい」→ 求人検索（既存）と同じ

**最も多いのは1と5**。ミサキは「なんとなく他の選択肢を見たい」段階。具体的な条件はまだ固まっていない。

**設計方針**: 初期画面は「あなたの近くの施設」（位置情報ベース）or「エリアから探す」のシンプルな2択。条件検索は「もっと絞る」ボタンの先。

**優先度**: この視点自体がUI設計の根幹。

---

### 10. 求人検索との住み分けは？（求人=募集あり、施設=募集なくても見れる）

**明確な住み分け**:

| | 求人検索 | 施設検索 |
|---|---|---|
| リッチメニュー | 「求人を探す」 | 「施設を調べる」 |
| 表示データ | jobs + 紐付きfacilities | facilities（全件） |
| 求人なし施設 | 非表示 | 表示。「逆指名できます」CTA |
| ソート | 給与/新着順 | 距離/病床数/看護師数 |
| ゴール | 応募 | 情報収集 → LINE相談 |

**事業上の価値**:
- 求人検索: 即コンバージョン狙い。「今すぐ転職したい」層
- 施設検索: ナーチャリング。「まだ転職は考えてないけど情報だけ」層 → 将来のLINE登録
- 逆指名導線: 施設検索から「この施設について相談する」→ handoff → 病院に直接交渉。**これが手数料10%モデルの真価**

**設計方針**: リッチメニューで明確に分離。施設検索の全カードに「この施設について聞く」ボタンを配置。

**優先度**: ★★★（住み分けは明確にすべき。混在させるとUXが破綻する）

## B. 表示と導線（10視点）

### 11. 検索結果の表示形式（カルーセル？リスト？地図？）

**LINEの制約**:
- Flex Messageカルーセル: 最大12バブル。1バブルに施設1件。スワイプで横にめくる
- Flex Messageリスト: 縦並び。1メッセージに最大3件表示（それ以上はスクロールしにくい）
- 地図: LINE内ブラウザでGoogle Maps埋め込みは可能だが、遷移が重い

**推奨**: **Flex Messageカルーセル（最大10件）+ 「もっと見る」ボタン**。理由:
- ミサキはスマホでサクサク見たい。カルーセルはスワイプが直感的
- 10件超は「条件を絞って再検索」に誘導（24,488件を全部見せる必要はない）
- 地図はWebブラウザ版（LP側）で対応。LINE Bot内では不要

**設計方針**: カルーセル形式。1バブル = 1施設カード。

---

### 12. 施設カードに何を表示するか

**1バブルに収める情報（優先順位）**:

```
┌─────────────────────────┐
│ 🏥 東京都済生会中央病院    │ ← 名前（短縮版）
│ 📍 港区三田 | 赤羽橋駅 3分  │ ← 所在地 + 最寄駅
│ 🛏 523床 | 👩‍⚕️ 常勤641名    │ ← 規模感
│ 循環器/消化器/救急/産婦...  │ ← 主要診療科（5つまで）
│ 急性期 | DPC標準病院群      │ ← sub_type + dpc_group
│ ─────────────────────── │
│ [求人3件あり]  [詳しく聞く] │ ← CTAボタン
└─────────────────────────┘
```

**表示しないもの**: 住所の番地（不要）、setter_type（看護師に興味ない）、source（内部情報）、last_synced_at

**注意**: nurse_fulltimeがNULLの場合（クリニック等）は「看護師数」行を非表示にする

---

### 13. 求人ありの施設を優先表示すべきか？

**結論: Yes。絶対に優先すべき。**

理由:
- 事業のゴールは「紹介して成約」。求人がある施設が最優先
- `has_active_jobs = 1` の施設を先頭、`0` を後方に
- ただし求人なし施設も表示する（逆指名導線のため）

**ソート順の提案**:
1. `has_active_jobs DESC`（求人あり優先）
2. `active_job_count DESC`（求人数多い順）
3. 距離 or 病床数（ユーザー選択）

---

### 14. 「この施設の求人を見る」ボタン → D1 jobsとの紐付け

**現状**: 紐付け率4.3%（名前完全一致のみ）。これでは使い物にならない。

**改善案**:
1. **施設名の正規化マッチング**: 法人名を除去して比較（「医療法人社団○○会」を削除）
2. **住所マッチング**: 同じ市区町村+施設名部分一致
3. **手動マッピングテーブル**: 主要施設は手動で紐付け

**実装の現実**:
```sql
-- facility_id が設定済みのjobsだけJOIN
SELECT f.*, j.title, j.salary_min, j.salary_max 
FROM facilities f 
LEFT JOIN jobs j ON j.facility_id = f.id AND j.status = 'active'
WHERE f.address LIKE '%横浜市%'
```

**紐付け率向上タスク**: jobsテーブルの`facility_id`を埋めるバッチ処理が先決。名前の部分一致+市区町村一致で8-15%まで上がる見込み。

**優先度**: ★★★（紐付けなしでは施設検索→求人の導線が機能しない）

---

### 15. 「この施設について聞く」ボタン → handoff

**これが最重要CTA。**

フロー:
1. ユーザーが施設カードの「この施設について聞く」をタップ
2. LINE Botが「○○病院について、どんなことが気になりますか？」と質問
3. ユーザーが回答（「給与」「夜勤回数」「人間関係」等）
4. Bot: 「承知しました。担当の転職アドバイザーが○○病院の詳細をお調べして折り返しますね」
5. → handoff → Slack通知 → YOSHIYUKI対応

**逆指名リクエストとの統合**: 
```
trackFunnelEvent('REVERSE_NOMINATION', userId, { facility: facilityName, area: city })
```

**設計方針**: ボタンのpostback dataに施設IDを含める → worker.jsで施設情報を取得 → handoffフローに入る

---

### 16. 検索結果が多すぎる場合のページング

**データ量の現実**:
- 「横浜市」で検索 → 病院 ~200件 + クリニック ~5,000件
- 全部表示は不可能。LINEのFlex Messageは1送信5バブルが体感限界

**設計方針**:
- 1回の表示: 5件（カルーセル）
- 「次の5件を見る」ボタン → offset + 5
- 20件（4ページ）見ても見つからない → 「条件を追加しませんか？」提案
- **category別タブは不要**。デフォルトで病院優先、「クリニックも見る」で切替

**技術実装**: KVにページング状態を保存
```json
{ "search_offset": 5, "search_query": { "city": "横浜市", "category": "病院" } }
```

---

### 17. 検索結果が0件の場合

**発生パターン**:
- 条件が厳しすぎる（「横浜市+循環器+500床以上+駅徒歩3分以内」→ 0件）
- 入力ミス（「よこはま」→ ひらがなでヒットしない）

**対応**:
1. 「条件に合う施設が見つかりませんでした」
2. 条件緩和の提案: 「○○を外すと△件見つかります」（条件を1つずつ外して件数を表示）
3. 「お探しの条件を担当者に伝えて、ぴったりの施設を探してもらいませんか？」→ handoff

**設計方針**: 0件 = handoff導線のチャンス。ネガティブ体験にしない。

---

### 18. 検索履歴の保存は必要か？

**結論: 初期実装では不要。**

理由:
- LINE Botのセッション（KV）に会話履歴が残る。検索条件は会話から復元可能
- 「前に調べた施設をもう一度見たい」→ 施設名を言えば再検索できる
- 検索履歴UIの開発コスト vs 使用頻度 → 見合わない

**将来**: LINE Sessions KVに `last_searches` を保存。リッチメニューの「履歴」タブで表示。ただしユーザー数が100名超えてからで十分。

---

### 19. 「お気に入り」機能は必要か？

**結論: 初期実装では不要。**

理由:
- ミサキは1-3施設を比較検討する段階。10個もお気に入りに入れない
- 「気になる施設」→ 「この施設について聞く」→ handoff の方がコンバージョンに直結
- お気に入り = 行動しない（保存して満足）。「聞く」= 行動する。後者を促すべき

**将来**: handoff済みの施設を「相談中リスト」として自動保存する方が有用。

---

### 20. Google Mapsへのリンクは有効か？

**結論: Yes。施設カードに地図リンクを入れるべき。**

理由:
- `lat`, `lng` が全件揃っている → `https://maps.google.com/?q={lat},{lng}` で即生成
- ミサキは「この病院どこにあるの？」→ 地図で確認 → 通勤経路を考える
- LINE内ブラウザでGoogle Mapsが開く。追加開発コスト0

**設計方針**: 施設カードのフッターに小さく「📍 地図を見る」リンク。メインCTAの邪魔にならない位置。

## C. 技術実装（5視点）

### 21. D1のFTS（全文検索）は使えるか？

**結論: 使えるが、不要。**

**D1 (SQLite) のFTS5**:
- `CREATE VIRTUAL TABLE facilities_fts USING fts5(name, departments, address)`
- 日本語トークナイザ: D1はicu tokenizerをサポートしていない。SQLite標準のascii/unicode61は日本語に対応していない
- つまり「循環器内科」で検索しても「循環器」でヒットしない（単語分割ができない）

**代替**: `LIKE '%keyword%'` で十分。24,488件のLIKE検索は50ms以下で完了する（後述 #22）。

**将来**: 検索精度が問題になったら、`trigram` tokenizerの検討。ただし現段階では過剰。

---

### 22. LIKE検索の性能（24,488件で遅くないか）

**結論: 問題なし。**

**根拠**:
- D1 (SQLite) はサーバーレスだがCPU上限あり。ただし24,488件は極めて小さいデータセット
- `LIKE '%keyword%'` はインデックスを使えないが、フルスキャンでも24,488行×数カラムなら10-50ms
- 既存のcountCandidatesD1()が同規模のクエリを実行済みで、問題なく動作している

**ベンチマーク（推定）**:
```
LIKE '%横浜市%' on address: ~20ms（24,488行スキャン）
LIKE '%循環器%' on departments: ~30ms（テキストが長い分やや遅い）
複合条件（3つのLIKE + bed_count）: ~50ms
```

**最適化が必要になる閾値**: 10万件超。現状の24,488件なら最適化不要。

### 23. 検索クエリのパース（「横浜 循環器 300床」をどう分解するか）

**LINE Botの場合**: 自由テキスト入力をパースする必要がある。

**パース戦略**:

```javascript
function parseSearchQuery(text) {
  const result = { city: null, department: null, bedMin: null, category: null, station: null };
  
  // 病床数: 数字+床
  const bedMatch = text.match(/(\d+)床/);
  if (bedMatch) result.bedMin = parseInt(bedMatch[1]);
  
  // カテゴリ: 病院/クリニック/訪問
  if (text.includes('病院')) result.category = '病院';
  if (text.includes('クリニック') || text.includes('診療所')) result.category = 'クリニック';
  
  // 市区町村: AREA_CITY_MAPの値リストから前方一致
  for (const cities of Object.values(AREA_CITY_MAP)) {
    for (const city of cities) {
      if (text.includes(city.replace('市', '').replace('区', ''))) {
        result.city = city;
        break;
      }
    }
  }
  
  // 診療科: 主要科目リストから部分一致
  const DEPT_KEYWORDS = ['内科','外科','整形外科','循環器','小児科','産婦人科',
    '精神科','リハビリ','眼科','皮膚科','耳鼻','泌尿器','脳神経','救急'];
  for (const dept of DEPT_KEYWORDS) {
    if (text.includes(dept)) result.department = dept;
  }
  
  return result;
}
```

**問題点**: 自然言語のあいまい性（「大きい病院」= 何床？）→ 確認ステップを入れる。

**代替案**: パースせずにAI（GPT-4o-mini）に構造化させる。既にworker.jsでOpenAI呼び出しの基盤がある。ただしレイテンシが1-2秒増える。

**推奨**: ボタン選択UI（段階的絞り込み）をメインにし、自由テキストパースはフォールバック。

### 24. facilitiesとjobsのJOINクエリの設計

**現状の問題**: `jobs.facility_id` がほとんど埋まっていない（紐付け率4.3%）。

**JOINクエリ（紐付け済みの場合）**:
```sql
-- 施設一覧 + 求人数
SELECT f.id, f.name, f.category, f.sub_type, f.address, 
       f.nearest_station, f.station_minutes, f.bed_count, 
       f.departments, f.nurse_fulltime, f.dpc_group,
       COUNT(j.id) as job_count
FROM facilities f
LEFT JOIN jobs j ON j.facility_id = f.id AND j.status = 'active'
WHERE f.address LIKE '%横浜市%'
GROUP BY f.id
ORDER BY job_count DESC, f.bed_count DESC
LIMIT 10 OFFSET 0
```

**紐付けなしのフォールバック**: `has_active_jobs` と `active_job_count` カラムを活用（バッチ更新）。

**紐付け率改善のバッチSQL**:
```sql
-- Step 1: 法人名を除去した施設名で紐付け
UPDATE jobs SET facility_id = (
  SELECT f.id FROM facilities f 
  WHERE jobs.title LIKE '%' || f.name || '%'
  OR f.name LIKE '%' || REPLACE(jobs.title, '看護師', '') || '%'
  LIMIT 1
) WHERE facility_id IS NULL;
```

**優先度**: 紐付けバッチを先に走らせること。JOINクエリ自体はシンプル。

### 25. 新しいインデックスは必要か？

**既存インデックス**:
- `idx_facilities_prefecture` (prefecture)
- `idx_facilities_lat_lng` (lat, lng)
- `idx_facilities_category` (category)
- `idx_facilities_has_jobs` (has_active_jobs)
- `idx_jobs_facility` (facility_id)
- `idx_jobs_status` (status)
- `idx_jobs_work_style` (work_style)

**追加が必要なインデックス**:

```sql
-- 1. 市区町村検索の高速化（cityカラムがある場合）
CREATE INDEX IF NOT EXISTS idx_facilities_city ON facilities(city);

-- 2. 複合インデックス（エリア+カテゴリ+求人有無: 最頻出クエリパターン）
CREATE INDEX IF NOT EXISTS idx_facilities_pref_cat_jobs 
ON facilities(prefecture, category, has_active_jobs);

-- 3. 病床数ソート用
CREATE INDEX IF NOT EXISTS idx_facilities_bed_count ON facilities(bed_count);

-- 4. 駅名検索用（LIKE '%keyword%' には効かないが、前方一致なら有効）
-- → 不要。LIKE '%駅名%' ではインデックスが使えない。24,488件なら全件スキャンで十分

-- 5. jobsの紐付け改善用
CREATE INDEX IF NOT EXISTS idx_jobs_title ON jobs(title);
```

**結論**: `idx_facilities_city` と `idx_facilities_pref_cat_jobs` の2つを追加すれば十分。24,488件規模ではインデックス過剰は逆にINSERT/UPDATE性能を落とすだけ。
