# AI履歴書作成機能 — 25視点の深掘り設計分析

> 2026-04-06 作成
> 対象: worker.js career_sheet / buildTemplateResume / buildResumeConfirmMessages

---

## 現状整理

### 既存の実装

```
フロー: apply_consent → career_sheet → apply_confirm → interview_prep → handoff
```

- `generateAnonymousProfile(entry)`: 病院確認用の匿名プロフィール（career_sheetフェーズで使用）
- `buildTemplateResume(entry)`: テンプレート穴埋め式の職務経歴書ドラフト（AI不使用版）
- `buildResumeConfirmMessages(entry, env)`: OpenAI GPT-4o-miniでテンプレートを改善し、LINEで送信
- 入力済みデータ: area, workStyle, facilityType, department, urgency, qualification, experience, workplace, strengths, change, concern, workHistoryText, name

### 2つの生成物が混在している
1. **匿名プロフィール**（career_sheet）: 病院に打診するための資料。名前なし
2. **職務経歴書ドラフト**（buildResumeConfirmMessages）: Q-flow完了後に生成。AI改善あり

→ 「AI履歴書作成」機能は、この2つ目を独立した価値提供として進化させる話。

## A. 何を作るのか（8視点）

### 視点1: 「履歴書」と「職務経歴書」の違い。看護師が必要なのはどちら？

| | 履歴書 | 職務経歴書 |
|--|--------|-----------|
| 目的 | 基本情報・学歴・職歴の一覧 | 業務内容・スキル・実績の詳細 |
| 形式 | JIS規格フォーマット（写真貼付） | A4自由形式 1-2枚 |
| 看護師の重要度 | 必須（提出義務） | **転職では超重要**（差がつくのはここ） |

**結論**: 両方必要だが、AIが価値を出せるのは**職務経歴書**。
- 履歴書: 事実の羅列。AIの出番は志望動機・自己PRの文面生成のみ
- 職務経歴書: 経験の言語化が苦手な看護師が多い。**ここがAIの主戦場**

**推奨**: 「職務経歴書」をメイン機能とし、履歴書は志望動機・自己PRのテキスト生成に限定。

### 視点2: 看護師の職務経歴書に必要な項目

```
【必須項目】
1. 保有資格（看護師/准看護師/保健師/助産師/認定看護師/専門看護師）
2. 経験年数（総年数 + 科別の内訳）
3. 所属科の履歴（内科/外科/救急/ICU/オペ室/透析/訪問看護 等）
4. 担当業務の詳細
   - 病棟: 受持患者数、重症度、夜勤回数
   - 外来: 1日の患者数、処置内容
   - 訪問: 担当件数、対象疾患
5. 使用可能な医療機器・看護技術
6. 研修・委員会活動
7. プリセプター/リーダー経験
8. 志望動機
9. 自己PR

【あると差がつく項目】
10. 具体的な成功エピソード（退院支援、急変対応等）
11. 学会発表・院内発表
12. 電子カルテシステムの使用経験（種類）
```

**現状のギャップ**: 現在のentry（KVデータ）では2,3,4,5,6,7,10,11,12が未取得。
intake_lightで聞いているのはarea/workStyle/urgencyの3問のみ。Q-flowでもqualification/experience/workplace/strengths/change/concernまで。

→ 職務経歴書の品質を上げるには**追加ヒアリングが必須**。

### 視点3: AIが生成するのは全文か、テンプレートの穴埋めか？

**現状**: ハイブリッド方式
1. `buildTemplateResume()` でテンプレート穴埋め（AI不使用版をフォールバック）
2. OpenAI GPT-4o-miniでテンプレートを改善（500文字以内制限）

**推奨**: 3段階方式

| レベル | 方式 | 入力量 | 品質 |
|--------|------|--------|------|
| Lv.1 速攻版 | テンプレート穴埋め | intake_light 3問だけ | 低（概要のみ） |
| Lv.2 標準版 | AI生成（現行） | Q-flow 10問 | 中（使えるドラフト） |
| Lv.3 本格版 | AI対話型改善 | 追加ヒアリング5-7問 | 高（そのまま提出可能レベル） |

**Lv.3が差別化ポイント**。ただし入力コストとの天秤。

### 視点4: 生成された文書の品質基準

```
【品質チェックリスト】
□ 敬語が統一されている（です・ます調 or である調）
□ 医療用語が正確（例: 「バイタル」ではなく「バイタルサイン」）
□ 誇張表現がない（「数百人の患者」→具体的数字が必要）
□ 個人情報が漏れていない（入力されていない情報を捏造しない）
□ 看護師の一般的なキャリアパスと矛盾しない
□ 500文字以内（LINE送信用）/ A4 1-2枚相当（PDF版）
□ コピペしてそのまま使える文体
```

**最大のリスク**: AIが入力されていない経験を「それっぽく」捏造すること。
→ `buildTemplateResume()`のフォールバック（未入力は「未入力」と明記）は正しい設計。
→ AIプロンプトに「入力されていない情報は絶対に補完するな。（未入力）と表記せよ」を明記必須。

### 視点5: ユーザーが入力すべき情報と、AIが補完する情報の境界

| 情報 | 入力者 | 理由 |
|------|--------|------|
| 氏名 | ユーザー | 個人情報 |
| 資格 | ユーザー（選択式） | 間違えられない |
| 経験年数 | ユーザー（選択式） | 間違えられない |
| 所属科の履歴 | ユーザー（選択式+自由記述） | 事実情報 |
| 担当業務の詳細 | ユーザー（自由記述） | 本人しか知らない |
| 志望動機 | **AI生成** | 転職理由+応募先の特徴からAIが構成 |
| 自己PR | **AI生成** | 強み+経験からAIが構成 |
| 文書の構成・文体 | **AI** | テンプレートに従って整形 |

**核心**: 事実情報はユーザー入力。**表現・構成・文体がAIの担当領域**。
看護師は「自分の経験をうまく言葉にできない」のが最大の悩み。ここをAIが解決する。

### 視点6: 1回の生成で完成するか、何回かの修正ラウンドが必要か？

**現状**: 1回生成 →「OK！これでいい」or「修正したい」の2択。修正したい場合はhandoffに飛ぶ。

**課題**: 「修正したい」→ handoff は雑すぎる。
- ユーザーは「志望動機だけ変えたい」のに全部やり直し
- 担当者の手間が増える

**推奨**: LINE内で2回まで修正ラウンド

```
Round 1: AI生成 → 確認
  ├─ OK → 次へ
  └─ 修正 → 「どの部分を修正しますか？」
      ├─ 志望動機
      ├─ 自己PR
      ├─ 経歴の詳細
      └─ 全部書き直し

Round 2: 修正版生成 → 確認
  ├─ OK → 次へ
  └─ もう1回修正 → Round 3（最終）or handoff
```

2回で収束しなければhandoff。3回以上のループはUX崩壊。

### 視点7: 生成結果の形式（テキスト？PDF？画像？）

| 形式 | メリット | デメリット | 実装コスト |
|------|---------|-----------|-----------|
| テキスト（現行） | LINE内で即閲覧 | コピペが面倒、見栄え悪い | ゼロ（済） |
| PDF | そのまま印刷・提出可能 | LINE内プレビュー不可、生成コスト | 中（Playwright or サーバーサイド） |
| 画像 | 見栄えが良い、LINE内表示可 | 編集不可、文字認識必要 | 中（Playwright） |
| Flex Message | LINE内で見栄え良く表示 | LINE外で使えない | 低 |

**推奨**: 2段階
1. **即時**: テキスト（LINE内で確認・修正用）← 現行
2. **確定後**: PDF生成してLINEで送信（提出用）← 新規追加

PDF生成はPlaywrightが既にカルーセル画像生成で稼働中なので、HTML→PDF変換は低コストで実装可能。

### 視点8: LINEの文字数制限（5,000文字）で職務経歴書は収まるか？

```
LINE Message API テキストメッセージ: 最大5,000文字
LINE Reply API: 最大5メッセージ/回

職務経歴書の一般的な文字数:
- 簡易版: 500-800文字（現行の500文字制限）
- 標準版: 1,000-2,000文字
- 詳細版: 2,000-3,000文字

看護師3年目: 約800文字で十分
看護師10年目（複数科経験）: 1,500-2,500文字
```

**現行500文字制限は短すぎる**。経験豊富な看護師の経歴書は収まらない。

**対策**:
1. LINE内プレビュー: 500文字の要約版（現行維持）
2. 全文: 2メッセージに分割送信（splitText関数が既に存在）
3. 提出用: PDF（5,000文字制限を回避）

## B. ユーザー体験（8視点）

### 視点9: 履歴書作成の前提条件（求人検索済みか？handoff前か？）

**現状のフロー位置**:
```
intake_light → matching → apply_info(名前入力) → apply_consent → career_sheet → apply_confirm → handoff
```

career_sheetは「応募同意後、handoff前」。つまり応募先が決まった状態で生成される。

**問題**: この位置だと「履歴書作成」は応募プロセスの一部であり、**独立した価値提供になっていない**。

**推奨**: 2つのエントリーポイントを用意

| パターン | タイミング | 目的 |
|---------|-----------|------|
| A. 応募フロー内（現行） | apply_consent後 | 応募先に合わせた志望動機生成 |
| B. 独立機能（新規） | リッチメニューからいつでも | 「まだ応募しないけど準備したい」需要 |

パターンBが**LINE登録の動機**になる。「友だち追加で職務経歴書を無料作成」。

### 視点10: 入力ステップ数と離脱率の関係

```
一般的なLINE Botの離脱率:
- 1-3問: 離脱率 10-20%
- 4-6問: 離脱率 30-40%
- 7-10問: 離脱率 50-70%
- 10問超: 離脱率 70%以上

現在のQ-flow FULL: 10問 → 推定離脱率60%以上
```

**推奨**: 最小入力で「まず1回出力を見せる」

```
Step 1: 資格（Quick Reply 1タップ）
Step 2: 経験年数（Quick Reply 1タップ）
Step 3: 直近の所属科（Quick Reply 1タップ）
→ ここで一旦ドラフト生成（3タップ、30秒）

Step 4: 「もっと詳しくしますか？」
  → はい → 追加質問（担当業務、強み等）
  → これでいい → 完了
```

**3タップで最初の出力を見せる**ことが離脱防止の鍵。
intake_lightの3問（area/workStyle/urgency）とは別に、経歴書用の3問を設計する。

### 視点11: テキスト入力が必要な項目のUX

**テキスト入力が必要な項目**:
1. 氏名（apply_infoで取得済み）
2. 職務経歴の詳細（「どんな業務を担当していましたか？」）
3. 志望動機の元ネタ（「転職で大切にしたいことは？」）

**課題**: LINEでの自由テキスト入力は離脱率が極めて高い。
- スマホのフリック入力で100文字以上は苦痛
- 「何を書けばいいかわからない」で固まる

**対策**:
```
悪い例: 「職務経歴を入力してください」
良い例: 「直近の職場で、1日の業務の流れを教えてください。
  例: 朝の申し送り→バイタル測定→点滴管理→記録→カンファレンス」

悪い例: 「自己PRを書いてください」  
良い例: 「患者さんやご家族に感謝された経験はありますか？
  簡単にで大丈夫です。例: 退院時に手紙をもらった」
```

**具体例を必ず提示**。回答のハードルを下げる。

### 視点12: Quick Replyだけで入力できる項目とテキスト必須の項目

| 項目 | Quick Reply | テキスト必須 |
|------|------------|-------------|
| 資格 | ✅ 看護師/准看護師/保健師/助産師 | |
| 経験年数 | ✅ 1-3年/3-5年/5-10年/10年以上 | |
| 所属科 | ✅ 内科/外科/救急/ICU/小児/産婦/精神/訪問 | |
| 施設種別 | ✅ 急性期病院/慢性期/クリニック/訪問看護 | |
| 担当患者数 | ✅ 5人以下/5-10人/10-15人/15人以上 | |
| 夜勤回数 | ✅ なし/月4回以下/月5-8回/月9回以上 | |
| リーダー経験 | ✅ あり/なし | |
| プリセプター経験 | ✅ あり/なし | |
| 転職理由 | ✅ 既存選択肢（6種） | |
| 強み | ✅ 既存選択肢（7種） | |
| 職務経歴の詳細 | | ✅（自由記述） |
| 具体的エピソード | | ✅（自由記述） |
| 志望動機の補足 | | ✅（自由記述） |

**Quick Replyだけで10項目取得可能**。テキスト入力は最大3箇所に限定できる。
さらに、テキスト入力は**任意**にして、未入力ならAIが選択肢の情報だけで生成する。

### 視点13: 途中保存は必要か？

**現状**: KVにentryとして全状態保存済み。24時間で期限切れ（localStorage）。
Worker側のKV（Cloudflare）には永続保存。

**結論**: 途中保存は**既に実現している**。
- ユーザーがLINEを閉じても、KVのentryにphaseと入力済みデータが残る
- 翌日LINEを開いて何かメッセージを送れば、中断地点から再開可能

**追加で必要なこと**:
- 24時間以上経過した場合のリマインド（「職務経歴書の作成、続きからできますよ」）
- これはナーチャリングメッセージに組み込める

### 視点14: 生成中の待ち時間の体験

**OpenAI GPT-4o-mini応答時間**: 2-5秒（トークン数依存）

**現状**: 特に待機UXなし。ユーザーは白い画面を見て待つ。

**推奨**:
```
1. 即時レスポンス: 「職務経歴書を作成中です... 10秒ほどお待ちください」
2. ローディングアニメーション（LINE Flex Messageでは不可）
3. 代替: 先に「作成のポイント」を送信し、AIが完成したら別メッセージで送信

実装案:
- Reply APIで「作成中」メッセージを即返す
- Push APIでAI生成結果を送信（Reply APIは1リクエスト1回なので）
```

**注意**: Push APIはLINE公式アカウントの無料枠（月200通）を消費する。
ただし、履歴書作成まで到達するユーザーはハイインテントなので、Push API消費は許容範囲。

### 視点15: 「修正したい」のフロー

**現状**: resume=edit → handoff（担当者に丸投げ）

**改善案**:
```
「修正したい」
  ↓
「どの部分を修正しますか？」
  [志望動機] [自己PR] [経歴の詳細] [全体的に] [担当者に相談]
  ↓
（志望動機の場合）
「現在の志望動機:
 ○○○○○○○○

 どのように変えたいですか？
 例: もっと○○を強調したい / ○○のエピソードを入れたい」
  ↓
 ユーザーがテキスト入力
  ↓
 AIが修正版を生成
  ↓
「修正しました。こちらでよろしいですか？」
  [OK] [もう一度修正] [担当者に相談]
```

**実装コスト**: OpenAI API呼び出し追加（1修正あたり約0.5円）。ユーザー体験の改善効果を考えれば安い。

### 視点16: handoffとの連携

**現状**: career_sheet → apply_confirm → handoff の流れで、生成された経歴書はentry.careerSheet / entry.resumeDraftに保存。Slack通知に含まれる。

**改善点**:
1. **Slack通知にPDF添付**: 担当者がすぐ確認できる
2. **修正履歴の共有**: 「ユーザーは志望動機を2回修正した」→ 面談時の参考情報
3. **応募先との紐付け**: 「この経歴書はA病院向けに生成された」を明記
4. **担当者が経歴書を編集してユーザーに返す機能**: `!resume edit`コマンド

## C. コストとリスク（5視点）

### 視点17: OpenAI APIコスト

```
GPT-4o-mini料金（2026年4月時点）:
- Input: $0.15 / 1M tokens
- Output: $0.60 / 1M tokens

1回の履歴書生成:
- Input: プロンプト（~500 tokens）+ ユーザー情報（~200 tokens）= ~700 tokens
- Output: 経歴書ドラフト（~500文字 ≈ 400 tokens）
- コスト: (700 × 0.15 + 400 × 0.60) / 1,000,000 = 約0.035円

修正1回あたり:
- 追加コスト: 約0.03円

1ユーザーあたり（生成1回 + 修正2回）:
- 合計: 約0.1円

月100人が使っても: 約10円
```

**結論**: コストは完全に無視できるレベル。GPT-4o-miniは安すぎる。
仮にGPT-4oに上げても1回あたり約3円。月300円。問題なし。

### 視点18: 個人情報の取り扱い

**現在保存されている個人情報**:
- 氏名（apply_infoで入力）
- 電話番号（handoff_phone_number）
- LINE userId（自動取得）

**保存先**: Cloudflare KV（Workers KV）
- 暗号化: Cloudflareのインフラレベル暗号化（at rest）
- データ保持期間: KVに明示的なTTL設定なし → 永続

**リスク**:
1. KVのデータがCloudflareダッシュボードから閲覧可能（管理者アクセス）
2. OpenAI APIに個人情報を送信している（名前は送っていないが経歴は送っている）
3. プライバシーポリシーに「AI処理」の記載があるか要確認

**対策**:
- 履歴書生成時のOpenAI APIには名前・電話番号を**送信しない**（現行は送っていないので問題なし）
- KVのTTL設定: 応募完了後90日で自動削除
- プライバシーポリシーに「AIによる文書生成のためにお客様の経歴情報を外部AIサービスに送信します」を追記

### 視点19: 生成された履歴書の正確性の保証

**リスクシナリオ**:
1. ユーザーが嘘の経歴を入力 → AIがそれをもとに経歴書を生成 → 病院に提出
2. AIが入力情報を拡大解釈 → 「内科3年」→「急性期内科病棟で重症患者を担当」（事実でない可能性）
3. AIが存在しない資格や経験を追加

**対策**:
```
1. 免責表示（必須）:
   「この職務経歴書はAIが入力情報をもとに作成したドラフトです。
    内容に誤りがないかご自身で必ず確認してください。
    記載内容の正確性はご本人の責任となります。」

2. AIプロンプトの制約:
   - 「入力された情報のみを使え。推測で情報を追加するな」
   - 「具体的な数字がない場合は○○を入れるな」
   - 「資格は入力されたもののみ記載」

3. 確認フロー:
   - 生成後に必ずユーザー確認を挟む（現行通り）
   - 「内容は正確ですか？」の明示的な同意
```

### 視点20: 悪用リスク

| リスク | 可能性 | 対策 |
|--------|--------|------|
| 大量生成（API負荷） | 低（LINE経由なのでレート制限あり） | 1ユーザー1日3回まで |
| 他人になりすまし | 低（LINE認証済み） | — |
| 架空の経歴で作成 | 中 | 免責表示+確認フロー |
| 競合の偵察 | 低 | 実害なし |

**LINE Botの構造上、悪用リスクは極めて低い**。
- 1 LINEアカウント = 1ユーザー（複垢作成のコストが高い）
- APIコストが安いので大量生成されても金銭的ダメージは微小

### 視点21: 法的リスク

**職業安定法上の論点**:
- 有料職業紹介事業者がAIで書類作成を代行すること自体は禁止されていない
- ただし「AIが作成した」ことを秘匿して病院に提出するのは問題になりうる

**消費者保護の観点**:
- AIが生成した文書であることの明示義務（景品表示法の類推）
- 「AIが作成したから品質保証」のような誤認表現は不可

**個人情報保護法**:
- AIサービス（OpenAI）への個人情報送信には利用目的の通知が必要
- OpenAIのAPI利用規約上、APIデータはモデル学習に使用されない（Data Processing Addendum）

**推奨対応**:
1. 利用規約に「AI生成文書の免責」条項を追加
2. プライバシーポリシーに「外部AIサービスの利用」を追記
3. 生成文書に「AIドラフト」の注記を入れる
4. 病院への提出時は「求職者が確認・修正済み」と明記

## D. 差別化（4視点）

### 視点22: 大手転職サイトの履歴書作成機能と比較

| サービス | 方式 | 特徴 | 弱点 |
|---------|------|------|------|
| ナース人材バンク | テンプレートDL | PDF/Word形式 | 穴埋め式、AI非対応 |
| マイナビ看護師 | Web入力→PDF | 項目入力してPDF出力 | 志望動機は自分で書く |
| レバウェル看護 | 担当者が代筆 | 品質高い | 面談必須、時間かかる |
| 看護roo! | テンプレ+例文集 | 例文が豊富 | コピペ感が出る |
| ジョブメドレー | なし | — | — |

**神奈川ナース転職の差別化ポイント**:
1. **LINE完結**: ブラウザ移動不要。チャットだけで完成
2. **AI対話型**: 「何を書けばいいかわからない」をAIが解決
3. **3タップで初版**: 最短30秒で初回ドラフト
4. **看護師特化プロンプト**: 一般的なAI文書生成ではなく、看護師の職務経歴書に最適化
5. **修正がチャット**: 「志望動機をもっと積極的にして」で即修正

**大手にないもの**: AIによるリアルタイム対話型の書類作成。これは小規模だからこそ素早く実装できる。

### 視点23: AIだからこそできること

```
1. 看護師特化の文脈理解
   - 「ICU 3年」→ 「急性期医療における重症管理の経験」に自動変換
   - 「夜勤きつい」→ 「ワークライフバランスを重視したキャリア形成を希望」に言い換え

2. 応募先に合わせた志望動機の自動調整
   - A病院（急性期）向け: 「急性期での経験を活かし...」
   - Bクリニック（訪問看護）向け: 「地域に根差した看護を...」
   → 応募先のconfig.js情報と連携すれば自動生成可能

3. 弱みの言い換え
   - 「経験2年で浅い」→ 「柔軟な学習姿勢と新しい環境への適応力」
   - 「ブランク3年」→ 「育児経験を通じた共感力の向上」

4. 複数パターン生成
   - 「積極的バージョン」と「堅実バージョン」を出し分け
   - ユーザーが好みの方を選べる

5. 面接想定Q&Aの自動生成
   - 経歴書の内容から「面接で聞かれそうな質問」を予測
   - 回答例もセットで提示
   → interview_prepフェーズとの自然な接続
```

### 視点24: 「無料特典」としてのマーケティング効果

**現状の集客導線**:
```
TikTok/Instagram → LP(ミニ診断) → LINE登録 → マッチング
```

**履歴書作成を追加した場合**:
```
TikTok/Instagram → LP → LINE登録
                          ├─ 求人検索（従来）
                          └─ 職務経歴書を無料で作る（新規）← 登録動機の追加
```

**効果予測**:
1. **LINE登録率向上**: 「履歴書作成」は明確な価値。「求人を見る」よりハードルが低い
2. **SNSコンテンツのネタ**: 「AIが30秒で経歴書作ってくれた」→ TikTok/Reelsネタ
3. **SEO記事との連携**: 既存の`resume-tips.html`から「AIで作ってみる」への自然な導線
4. **リード獲得**: 経歴書作成 = 転職意欲の高いユーザーの特定（ナーチャリング対象）

**コピー案**:
```
LP: 「看護師の職務経歴書、AIが30秒で作成。LINE友だち追加で今すぐ無料で試す」
TikTok: 「経歴書書くの面倒すぎて放置してたけど、AIに3つ答えたら完成した」
Instagram: 「転職準備、まだ始めてない？ 30秒で経歴書ドラフトを作ろう」
```

### 視点25: 履歴書作成がhandoff率に与える影響の予測

**仮説**: 履歴書作成はhandoff率を**上げる**。

**理由**:
```
1. サンクコスト効果
   - 経歴書を作る = 転職準備に時間を投資した
   - 「せっかく作ったんだから応募してみよう」心理

2. 自信の醸成
   - 「自分の経歴、意外とちゃんとしてる」と気づく
   - AIが「あなたの強みは○○です」と言語化 → 自己肯定感UP

3. 障壁の除去
   - 「書類準備が面倒」は転職活動の最大障壁の一つ
   - これが解消されることで「あとは応募するだけ」状態になる

4. 応募先の具体化
   - 経歴書作成中に「この求人に合った志望動機」を見る
   - → 「この病院いいかも」と具体的に検討が進む
```

**数値予測**:
```
現状（推定）:
  LINE登録 → handoff率: 5-10%

履歴書機能追加後（予測）:
  LINE登録 → 経歴書作成: 20-30%（登録者の）
  経歴書作成 → handoff率: 30-50%（作成者の）
  
  LINE登録 → handoff率: 6-15%（全体）
  → 現状比 +20~50%の改善
```

**計測方法**:
- KVにphase遷移ログを追加（resume_started / resume_completed / resume_to_handoff）
- GA4カスタムイベント: `resume_generate`, `resume_complete`, `resume_to_apply`
