## 現状の整理

| 項目 | 現在の実装 |
|------|-----------|
| 収集情報 | 免許取得年/職場種類/配属先+業務/前職詳細/保有資格/転職理由/氏名 |
| AI送信 | OpenAI GPT-4o-mini API に資格・経験年数・職歴・強み・転職理由・不安を送信（氏名は送信していない） |

| 同意取得 | rm_resume_start時に簡易説明あり（「個人情報は求人紹介の目的でのみ使用」「担当者以外に共有しない」） |

## 1. 各専門家の発言

### 専門家1: 個人情報保護法の専門家

現状の最大の問題は、プライバシーポリシー第10条6項でAI処理の委託先を「Anthropic社のClaude」と記載しているが、実際にはOpenAI GPT-4o-miniに送信している点だ。これは利用目的の虚偽記載に該当し、個人情報保護法第21条（利用目的の通知・公表）違反となる。直ちに修正が必要。また、OpenAI APIへの送信は「委託」に該当する（OpenAIのAPIは個人情報保護法第27条5項1号の「委託」であり、第三者提供には当たらない）。ただし委託先管理義務（法第25条）として、OpenAIのデータ処理契約（DPA）の内容を確認し、記録を保持する必要がある。有料職業紹介事業者として、職業安定法第51条の2に基づく個人情報の適正管理義務も負っている。

### 専門家2: セキュリティエンジニア

コードを確認した結果、AIに送信されるプロンプト（worker.js 4518-4529行目）には氏名は含まれていない。これは良い設計だ。ただし以下の問題がある。(1) Slackへのhandoff通知（5166-5208行目）に氏名・電話番号・LINE userId・履歴書ドラフト全文・直近会話が平文で送信されている。Slackはクラウドサービスであり、チャンネルへのアクセス権管理が甘いと漏洩リスクがある。(2) Cloudflare KVのデータは暗号化されていない。KV自体はCloudflareの暗号化ストレージだが、アプリケーション層での暗号化は施されていない。(3) OpenAI APIはSOC2認証済みで、APIデータはモデル学習に使用されない（opt-out確認済み前提）。30日間のログ保持がある点は留意。

### 専門家3: UXデザイナー

rm_resume_start（worker.js 4386行目）の同意説明は簡潔で良い。ただし「AIがお手伝い」とは書いてあるが、「外部AIサービスにデータが送信される」ことは明示されていない。看護師は病院の個人情報研修を受けているため、「AIに送信」と聞くと警戒する人が多い。しかし離脱を防ぐためには、(1) 同意文を長くしすぎない（3行以内）、(2) 詳細はリンクで提供、(3) 「施設名は伏せてOK」の記載は安心感を与えており維持すべき。氏名入力（Q8、4454行目）の段階で再度「個人情報は求人紹介の目的でのみ」と表示しているのは適切。ミサキの視点では「鬼電されない」「情報が勝手に使われない」が最重要なので、その点を強調すべき。

### 専門家4: 事業責任者（平島禎之）の視点

個人情報は紹介事業の核だ。名前がなければ病院に紹介できない。ただし、名前を聞くタイミングは現在のQ8（最後の質問）で正しい。最初に聞くと離脱する。差別化ポイントとして「個人情報を大切にするサービス」は使える。大手は登録時に名前・電話・メールを全部取るが、うちはLINEで完結し電話しない。これは手数料10%と並ぶ差別化要素だ。ただし過度なセキュリティ投資はコストに見合わない。最小限の対応で法的リスクをゼロにしろ。許可番号（23-ユ-302928）を同意画面に表示するのは信頼性向上に効果的。

### 専門家5: 大手競合の個人情報対応調査

主要競合の比較結果: (1) レバウェル看護 — 登録時に氏名・電話・メール・住所を取得。登録完了後すぐに電話がかかる。プライバシーポリシーは詳細だがAI利用の記載なし。(2) ナース人材バンク — 同様に登録時に全情報取得。SMS+電話で連絡。(3) マイナビ看護師 — フォーム登録で氏名・電話・メール。AI利用の記載なし。大手はいずれも「登録＝全個人情報提供」のモデル。ナースロビーの「LINE完結・電話なし・段階的情報取得」は明確な差別化。ただし大手はISMSやPマーク取得済みが多い。小規模事業者はそれに代わる信頼シグナルが必要。

### 専門家6: ミサキ（28歳看護師）

前の転職サイトに登録したら、3社から同じ日に電話が来て本当に嫌だった。だから「電話しません」って書いてあるのは安心する。AIに経歴を送るのは...正直よく分からないけど、「名前は送らない」「施設名は伏せていい」って言われたら大丈夫かな。でも「外部のAIサービスに送信」って書かれたらちょっと怖い。「AIが文章を整えます」くらいの表現のほうが自然。一番気になるのは、私の情報がどこに残るのか。30日で消えるなら安心。あと許可番号があるなら見せてほしい。怪しいサイトかどうか判断する材料になる。

### 専門家7: LINE Bot開発者

技術的に以下の改善が可能。(1) 氏名のAI非送信: 現状でも実装済み（worker.js 4518-4529行目のプロンプトに氏名は含まれていない）。氏名はentry.fullNameに保存し、テンプレートにローカル差し込みしている。(2) KV TTL: 現在30日設定。職業安定法の記録保存義務（2年）との整合性を考えると、KVの30日はセッションデータとしては適切。紹介成立後の記録は別途管理すべき。(3) Slackへの氏名送信: handoff通知から氏名を除外し、KVのキーIDのみ通知→担当者がダッシュボードで確認する方式が理想だが、ダッシュボードがない現状ではSlackに送らざるを得ない。代替策としてDMで送信しチャンネルには非表示にする方法がある。(4) プライバシーポリシーリンク: Flex Messageでリンクボタンを表示可能。

### 専門家8: 弁護士（リスク管理）

法的リスクの優先順位を整理する。(1) 最優先: プライバシーポリシーの「Anthropic社のClaude」をOpenAI記載に修正。虚偽記載は個人情報保護委員会の勧告対象になりうる。(2) OpenAI APIへの送信は個人情報保護法第27条5項1号の「委託」に該当する。委託先の監督義務（法第25条）として、OpenAIのDPA/Terms of Serviceの確認記録を保持せよ。(3) 小規模事業者（取扱件数5000件以下）であっても2022年改正で全事業者が対象。特例はない。(4) Slack通知に個人情報を含めること自体は業務上必要な範囲であれば問題ないが、アクセス制御（チャンネルメンバーの限定）が必要。(5) 漏洩時の対応: 2022年改正で個人情報保護委員会への報告義務（法第26条）と本人通知義務が追加された。漏洩対応マニュアルの整備を推奨。

## 2. 論点ごとの合意事項

### 論点1: 氏名をAIに送信すべきか？

**合意: 現状維持（送信しない）。既に適切に実装されている。**

- worker.js 4518-4529行目のプロンプトに氏名は含まれていない
- 氏名はentry.fullNameに保存し、ローカルで差し込む設計
- この設計は「データ最小化の原則」に合致しており、変更不要

### 論点2: 同意取得のタイミングと方法

**合意: 現在のrm_resume_start時の簡易同意を強化。2段階同意に変更。**

- **ステップ1（rm_resume_start）**: 現在の説明文に「有料職業紹介事業 許可番号23-ユ-302928」を追加。プライバシーポリシーへのリンクを追加
- **ステップ2（rm_cv_q8 = 氏名入力時）**: 「氏名は病院への紹介時にのみ使用します。AIには送信しません。」を追加
- 同意文は3行以内を維持（UXデザイナー・ミサキの意見を尊重）

### 論点3: プライバシーポリシーの更新

**合意: 以下を修正・追記する。**

1. 第10条6項: 「Anthropic社のClaude」→「OpenAI社のGPT-4o-mini」に修正。Cloudflare Workers AIの記載は維持
2. 第10条に追記: 「AIサービスに送信されるデータには氏名・電話番号等の直接的な個人識別情報は含まれません。AIはテキストの構成・改善のみに使用されます」
3. 第10条に追記: 「OpenAI社のAPIデータポリシーに基づき、送信データはAIモデルの学習には使用されず、API経由のデータは最大30日間のログ保持後に削除されます」
4. 最終更新日を「2026年4月6日」に変更

### 論点4: データ保存期間

**合意: KV TTL 30日は適切。ただし注意点あり。**

- KVの30日TTLはLINEセッション（チャット進行中のデータ）としては適切
- 紹介成立後の求職者記録は職業安定法第32条の16に基づき2年間保存義務 → これはKVとは別管理（現状Slackログが実質的な記録）
- KV TTL設定のコード上の確認は必要（実装済みかどうか）

### 論点5: Slackに個人情報を流すリスク

**合意: 短期的には現状維持。中期的にはダッシュボード構築を検討。**

- 現状、Slackは唯一の担当者通知手段であり、氏名を含めないと業務が回らない
- **即時対応**: Slackチャンネル（C0AEG626EUW）のメンバーを平島禎之のみに限定し、パブリックチャンネルにしない
- **中期対応**: 氏名・電話番号はDMで送信し、チャンネルには匿名化したサマリのみ投稿する方式を検討
- LINE userIdの表示は業務上必要（`!reply`コマンドで使用）なので維持

### 論点6: OpenAI APIへの送信データの最小化

**合意: 現状の送信内容は適切。ただし1点修正。**

- 現在送信: 資格/経験年数/現在の職場/職歴/得意なこと/転職の背景/不安
- 氏名・電話番号・LINE userIdは送信していない（確認済み）
- **修正点**: 「不安」（concernLabel）は履歴書生成に不要。プロンプトから削除可能
- 「職歴」（workHistoryText）はユーザーが自由入力するため、施設名等が含まれる可能性がある → rm_cv_q3の説明で「施設名は伏せてOK（○○病院等）」を案内済みで、これは維持

### 論点7: ナースロビーの差別化

**合意: 「個人情報を大切にするサービス」として積極的に訴求する。**

- 具体的な差別化メッセージ:
  - 「電話しません。LINEで完結。」（既にLP・広告で訴求中）
  - 「名前はAIに送りません。」（新規追加）
  - 「登録不要。LINEで相談するだけ。」（既存）
  - 「有料職業紹介事業 許可番号23-ユ-302928」の表示（新規追加）
- LPの信頼セクションに追加可能
- 大手の「登録→即電話」モデルとの明確な差を打ち出す

## 5. 討議の総括

### 発見された問題（重要度順）

1. **プライバシーポリシーの虚偽記載**（P0）: 実際はOpenAI GPT-4o-miniを使用しているのに「Anthropic社のClaude」と記載。即時修正必須。
2. **AI非送信情報の未明記**（P0）: 氏名をAIに送信しない設計は良いが、ポリシーに明記されていない。
3. **同意取得の不足**（P1）: 許可番号・ポリシーリンクの未表示。

### 適切に実装されていた点

1. 氏名がAIプロンプトに含まれていない設計（worker.js 4518-4529行目）
2. KV TTL 30日の設定
3. 「施設名は伏せてOK」の案内
4. 氏名入力を最後の質問（Q8）に配置

### 結論

ナースロビーの履歴書機能は、技術的には適切なデータ最小化が実装されている。しかし、プライバシーポリシーの記載が実態と乖離しており、法的リスクがある。P0の2件（ポリシー修正）を今日中に対応し、P1の2件（UX改善）を今週中に対応すべき。総工数は約1.5時間で完了可能。
