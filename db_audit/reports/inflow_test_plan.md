## 5. ログ記録ルール

### 5.1 各テストケース必須記録項目

```
テストID: A{Agent番号}-{連番}（例: A3-027）
実行時刻: YYYY-MM-DD HH:MM:SS JST
カテゴリ: A〜J
対応修正#: 1〜20（複数可）
入口: meta_ad / LP / follow / blog / area_page / salary_check / direct
入力内容: ユーザーが送信したテキスト/タップしたボタン（全ステップ）
期待結果: このテストで確認すべきこと（1行）
実際結果: 実際のBot応答（要約+スクリーンショットID）
判定: PASS / FAIL / WARN
FAIL理由: （FAILの場合のみ。具体的に何が期待と違ったか）
備考: （気づき、想定外の挙動等）
```

### 5.2 ログファイル形式

- 各Agentが `db_audit/reports/test_log_agent{N}.jsonl` にJSONL形式で記録
- 1テスト1行。フィールド名は上記の通り
- マイクロマネージャーが各Agentのログを10件毎にレビュー

### 5.3 スクリーンショットルール

- FAIL判定は必ずスクリーンショットを保存（`db_audit/reports/screenshots/`）
- PASS判定でもカルーセル表示/給与表示/handoffメッセージは保存推奨

## 6. 合格基準

### 6.1 全体合格基準

| 指標 | 合格ライン |
|------|-----------|
| 全体PASS率 | **95%以上**（380/400件以上） |
| FAIL 0件必須の項目 | 緊急キーワード検出(#4)、非看護師除外(#14)、「病院に直接聞く」廃止(#13) |
| 致命的バグ | **0件**（下記定義参照） |

### 6.2 致命的バグの定義（1件でも即不合格）

1. 緊急キーワード（「死にたい」等）で即handoffが発火しない
2. 派遣/介護職/言語聴覚士の求人が看護師結果に混入
3. 電話番号がSlackに転送されない
4. handoff後にBOTが勝手に応答する
5. SQLインジェクションが成功する
6. 「病院に直接聞く」ボタンが表示される
7. セッションデータの他ユーザーへの漏洩

### 6.3 カテゴリ別合格基準

| カテゴリ | 合格ライン | 理由 |
|---------|-----------|------|
| A. 入口テスト | 100% | 流入経路の取りこぼしは致命的 |
| B. エリア選択 | 95% | 新規エリア追加のため |
| C. マッチング品質 | 90% | スコア順の微差は許容 |
| D. 10件上限+導線 | 100% | 担当者接続=売上直結 |
| E. handoffフロー | 100% | 人命に関わる可能性（緊急キーワード） |
| F. NLP | 85% | 自然言語解析の限界は許容 |
| G. 条件変更 | 95% | UXの根幹 |
| H. エッジケース | 80% | 攻撃的入力の一部は許容 |
| I. E2E統合 | 95% | 全体フロー整合 |

### 6.4 判定フロー

```
全Agent完了
  ↓
マイクロマネージャーA/B がログ精査（手抜き/判定甘すぎ検出）
  ↓
監督者が集計
  ↓
致命的バグ: 1件でもあれば → 全体FAIL → 即修正→再テスト
カテゴリ別未達: → 該当カテゴリ再テスト
全体95%未満: → FAIL項目分析→修正→該当ケース再テスト
全基準クリア: → 合格 → デプロイ承認
```
