# リッチメニューv2 — 25視点ユーザージャーニー分析

作成日: 2026-04-06
分析者: ユーザージャーニー設計専門エージェント

---

## 現状整理

```
リッチメニュー案:
┌─────────────────────────────────────┐
│     お仕事探しをスタート（既存）      │  ← 上段（全幅）
├───────────┬───────────┬─────────────┤
│ NEW       │ SEARCH    │ SUPPORT     │  ← 下段（3分割）
│ 本日の    │ 施設検索  │ 履歴書作成  │
│ 新着求人  │           │ （AI）      │
└───────────┴───────────┴─────────────┘

現行Worker: 4状態リッチメニュー切替（default/hearing/matched/handoff）
DB: facilities 24,488件 / jobs 2,936件（ハローワーク）
フロー: welcome → 5-6問タップ → マッチング → 担当者引き継ぎ
```

---

## A. ユーザーの状態別（5視点）

### 1. 初回訪問（広告から来た。まだ何も知らない）

**心理**: 「何これ？怪しくない？」「転職サイトまた登録するの面倒」「電話かかってこない？」

**リッチメニューとの接点**:
- 上段「お仕事探しをスタート」は適切。CTA明確で迷わない
- 「新着求人」は初回では意味が薄い。何が新着かの基準がない（比較対象がない）
- 「施設検索」は初回で使うか？ → 特定病院名を検索したい人は少数派。大半は「条件で探したい」
- 「履歴書作成」は初回では早すぎる。信頼関係ゼロの段階でAIに個人情報を預けるハードルは高い

**提案**: 初回は上段「お仕事探しをスタート」に集中させるべき。下段は目に入っても「後で使えるんだな」程度で良い。現状の3機能は初回ユーザーに対して過剰。

---

### 2. 2回目訪問（前回求人見たけど決めなかった）

**心理**: 「あの求人まだあるかな」「他にいいのないかな」「前の条件覚えてる？」

**リッチメニューとの接点**:
- **「新着求人」は2回目訪問者にこそ刺さる**。前回見た求人との差分が価値になる
- 「施設検索」は「あの病院どうだったっけ」と名前を覚えている場合に有効
- 「履歴書作成」はまだ早い。2回目はまだ情報収集段階

**不足している機能**: 「前回の結果を見る」「条件を変えて再検索」がない。2回目ユーザーが最もやりたいのは「前回の続き」。現在のWorkerはセッション保持しているが、リッチメニューからワンタップで戻れない。

**提案**: 「前回の求人を見る」または「条件を変更」ボタンが2回目ユーザーには最も価値が高い。

---

### 3. 本気モード（転職を決意。情報収集段階を超えた）

**心理**: 「もう辞める。早く次を見つけたい」「履歴書書かなきゃ」「面接対策も」

**リッチメニューとの接点**:
- 「お仕事探しをスタート」→ 既に求人は見た。これをまた押すのは回帰行動
- 「新着求人」→ 有効。毎日チェックする動機がある
- 「施設検索」→ 有効。特定施設を深掘りしたい
- **「履歴書作成（AI）」→ まさにこの段階で必要**。本気モードの看護師がLINEから離れずに書類準備できるのは大きな差別化

**提案**: 本気モードのユーザーにとってこの3機能は全て理にかなっている。ただし現状のworker.jsでは `career_sheet` フェーズはあるがリッチメニューから直接遷移できる導線が弱い。

---

### 4. 迷い中（転職するかどうかまだ悩んでいる）

**心理**: 「今の職場もう嫌だけど、転職して失敗したら…」「年収下がらない？」「他の人はどうしてるの？」

**リッチメニューとの接点**:
- 「お仕事探しをスタート」→ 名前が直接的すぎる。「求人を探す＝転職する」と決めていない人にはハードルが高い
- 「新着求人」→ 迷い中の人は「新着」に興味はない。「相場感」「比較」がほしい
- 「施設検索」→ 現職と比較したい可能性はある
- 「履歴書作成」→ 論外。迷っている人に「準備しろ」は逆効果

**不足している機能**:
- **「転職Q&A」「年収チェック」「みんなの声」的なコンテンツ機能**がない
- 現行Workerには `faq_advice` フェーズ（年収相場/夜勤と年収/有利な時期/バレずに活動/有給の見分け方）が既にある → リッチメニューからのアクセス導線がない

**提案**: 迷い中ユーザーを放置するとブロックされる。「役に立つ情報」を提供し続けることでエンゲージメントを維持し、本気モードへの転換を待つ。FAQ/転職コラムへの導線が必要。

---

### 5. handoff後（担当者を待っている間）

**心理**: 「いつ連絡くるの？」「待ってる間に他も見ておこう」「キャンセルしたいかも」

**リッチメニューとの接点**:
- 現行Workerは `RICH_MENU_STATES.handoff` に切り替わる（求人を見る/経歴書/FAQ/自由入力）
- **案の3機能はhandoff状態では不適切**。handoff後に「お仕事探しをスタート」を再度見せるのは混乱を招く

**不足している機能**:
- 「担当者への質問を残す」（非同期メッセージ）
- 「面接準備ガイド」
- 「進捗確認」

**提案**: handoff後は専用メニューに切り替わる設計が既にWorkerにある。これは維持すべき。リッチメニューv2はdefault/hearingフェーズのデザインに集中してよい。

---

## B. 時間帯別（5視点）

### 6. 夜勤明けの朝6-8時

**状態**: 極度の疲労。判断力低下。ベッドに倒れ込む直前。スマホは惰性で触る。

**行動パターン**: Twitterやインスタをぼーっとスクロール → たまたま転職広告が目に入る → LINEを開くかもしれないが、集中力はない

**リッチメニューに求められること**:
- **ワンタップで結果が出ること**。テキスト入力は無理。選択肢も3つが限界
- 「新着求人」は良い。タップ1回で今日の求人が見られるなら夜勤明けでも使える
- 「施設検索」は厳しい。検索条件を入力する気力がない
- 「履歴書作成」は論外。疲れ切った状態でキャリアの棚卸しはしない

**提案**: 夜勤明けには「受動的に受け取れるコンテンツ」が最適。プッシュ通知で新着を送るほうがリッチメニューより効果的。

---

### 7. 日勤の昼休み12-13時

**状態**: 45分〜60分の昼休み。食事しながらスマホ。周りに同僚がいる。

**行動パターン**: 転職活動を「見られたくない」。LINEなら自然。画面をチラ見しながら。

**リッチメニューに求められること**:
- **短時間で完結する操作**。5分以内に1つのタスクが終わること
- 「新着求人」→ 良い。30秒で確認できる
- 「施設検索」→ 特定病院名を打つだけなら良い。条件検索は時間がかかる
- 「履歴書作成」→ 昼休みに始める人はいない。中断されるリスクが高い

**提案**: 昼休みは「確認系」が最適。新着チェック、前回の結果確認、お気に入り確認。「作業系」（履歴書等）は不向き。

---

### 8. 夜勤前の準備中18-20時

**状態**: これから12時間の夜勤。気分は重い。「こんな生活嫌だ」の感情がピーク。

**行動パターン**: 転職への意欲が最も高まる時間帯。「今日こそ本気で探す」

**リッチメニューに求められること**:
- **感情に寄り添いつつ、行動を促すUI**
- 「お仕事探しをスタート」→ まさにこのタイミング。5問なら夜勤前にできる
- 「新着求人」→ 有効。「今日出た良い求人あるかな」
- 「施設検索」→ 「あの病院、看護師募集してるかな」
- 「履歴書作成」→ 夜勤前にやる人は少ないが、ゼロではない

**提案**: 18-20時は最もコンバージョンが高い時間帯の可能性。全機能がフルに活きる。この時間帯にプッシュ通知を送る施策と組み合わせるべき。

---

### 9. 深夜の夜勤中休憩2-4時

**状態**: 仮眠室で30分。眠いが眠れない。孤独感。

**行動パターン**: SNSをだらだら見る。転職を考える（感情的に）。でも行動はしない。

**リッチメニューに求められること**:
- **読むだけのコンテンツ**。操作は最小限
- 「新着求人」→ 見るだけなら良い
- 「施設検索」→ 深夜にやらない
- 「履歴書作成」→ 深夜にやらない

**提案**: 深夜帯は「情報収集」と「感情的な共感」がキー。転職コラムやFAQへの導線があると、ブロック率低下につながる。

---

### 10. 休日の昼間

**状態**: 時間がある。冷静。情報収集に最適な精神状態。

**行動パターン**: 複数の転職サイトを比較。じっくり条件を検討。履歴書を書くならこのタイミング。

**リッチメニューに求められること**:
- **全機能がフル活用される唯一の時間帯**
- 「施設検索」→ 休日だからこそ、じっくり検索したい
- 「履歴書作成」→ 休日の午後が最適タイミング
- 「新着求人」→ 見る

**提案**: 休日の昼間に向けて全機能を最適化するのは正しい。ただし看護師の「休日」は不規則（シフト制）であり、曜日ではなく個人の状態に依存する。

## C. 機能の必要性（5視点）

### 11. 「新着求人」は本当に毎日必要か？ハローワークの更新頻度は？

**ファクト**:
- ハローワーク求人は**火〜土の毎日更新**（前営業日受理分、午前4時頃反映）
- 月曜は更新なし（土曜に反映済み）
- 現在のD1 jobsは2,936件。毎日の新着は推定30-80件（全職種）、看護師フィルタ後は5-15件程度
- hellowork_to_d1.pyが毎朝06:30に自動更新している

**分析**:
- 毎日5-15件の新着は「本日の新着」として成立する
- ただし**日によっては0件の可能性もある**（月曜、祝日翌日）
- 0件表示は信頼を損ねる → 「直近3日の新着」にするか、0件時のフォールバック設計が必要

**結論**: 新着機能は有効だが「本日の」は厳密すぎる。**「最新の求人」**としてバッファを持たせるべき。0件時は「新着はありませんが、あなたの条件に合う求人はこちら」と既存マッチング結果にフォールバック。

---

### 12. 「施設検索」で何を検索するのか？名前？エリア？条件？

**ユースケース分析**:
1. **施設名検索**（「○○病院って募集してる？」）→ 最も直感的。24,488件のDBが活きる
2. **エリア検索**（「川崎市の病院」）→ 既存の5問フローと重複する
3. **条件検索**（「日勤のみ+クリニック+横浜」）→ 5問フローそのもの

**問題点**:
- LINEのリッチメニューから「施設検索」を押した後のUXが不明確
- テキスト入力UI → 看護師は施設名を正確に打てない（略称、旧名称）
- Flex Messageでの検索UIには限界がある

**結論**: 「施設検索」は**施設名検索に特化**すべき。エリア・条件検索は既存フローと重複する。実装としては「施設名を入力してください」→ あいまい検索 → Flexカルーセルで結果表示。ただし、NLP検索は既にWorkerに実装済み（47都道府県+21都市名キーワード検出）なので、施設名検索の追加実装が必要。

**代替案**: 「施設検索」ではなく「気になる病院を調べる」というラベルにして、施設名入力に誘導する。

---

### 13. 「履歴書作成」はどの段階で必要か？初回？handoff前？

**看護師の転職タイムライン**:
```
情報収集（1-3ヶ月）→ 応募決意（1-2週間）→ 書類準備（3-7日）→ 応募→面接→内定
```

**分析**:
- 履歴書が必要になるのは**応募決意後**。つまりマッチング完了→担当者相談後
- 初回訪問時に履歴書を作る人は**ほぼゼロ**
- ナースロビーのファネルでは、handoff後に担当者が書類を案内するのが自然な流れ
- ただし「履歴書が面倒で転職を先延ばしにしている層」は存在する → そこにAI作成は刺さる

**最適タイミング**:
1. matchedフェーズ（求人を見た後）で「この病院に応募するなら履歴書を準備しましょう」
2. handoffフェーズで担当者が案内する際に「AI作成もできますよ」

**結論**: リッチメニューの常設枠に置くほどの使用頻度はない。**matchedフェーズのリッチメニュー（4状態切替の3番目）に配置するのが最適**。defaultメニューには不要。

---

### 14. リッチメニューに入れるべきだが案にない機能は？

**優先度順**:

1. **「転職Q&A / よくある質問」** ★★★
   - 迷い中ユーザーの受け皿。ブロック防止に直結
   - 既にWorkerに5問のFAQ（年収相場/夜勤/時期/バレずに/有給）が実装済み
   - リッチメニューからワンタップでアクセスできない現状は損失

2. **「条件を変えて再検索」** ★★★
   - 2回目以降ユーザーの最大ニーズ
   - Workerには条件部分変更UI（エリア/施設/働き方を個別変更）が実装済み
   - リッチメニューに「条件変更」を入れるだけでリピート率が上がる

3. **「担当者に相談」（直接handoff）** ★★
   - 5問が面倒な人のショートカット
   - 「とりあえず話を聞きたい」層を取りこぼさない
   - Welcome Quick Replyの「まず相談したい」と同等の機能

4. **「お気に入り / 保存した求人」** ★
   - browsedJobIdsは既にDB管理されている
   - 複数回訪問ユーザーの利便性向上

---

### 15. リッチメニューに入れるべきでない機能は？

1. **「年収診断」（単体ボタンとして）** ✕
   - STATE.mdで「相場比較→求人件数の段階表示に転換」と方針変更済み
   - 厚労省データの年収チェックは精度問題あり → FAQ内の1項目で十分

2. **「友達紹介」** ✕
   - 現段階でユーザー0人。紹介する友達がいない
   - 事業確立後（成約10件以上）に検討

3. **「設定」「プロフィール編集」** ✕
   - LINEボットに設定画面は不要。条件変更で十分
   - エンジニア視点のUIを看護師に押し付けない

4. **「ブログ / コラム」（外部リンク）** ✕
   - LINEの外に出すとユーザーが戻ってこない
   - コンテンツはLINE内のFAQとして提供すべき

5. **「通知設定」** ✕
   - 現時点でプッシュ通知を送っていない
   - 送り始めてからブロック率を見て判断

## D. 競合比較（5視点）

### 16. レバウェル看護のLINEは何を提供している？

**確認できた情報**:
- LINE公式アカウント: 「レバウェル看護（情報配信）」として存在
- 主な機能: 求人のお知らせ配信（週1回程度）、キャリアアドバイザーとのチャット
- リッチメニュー: 求人検索、転職相談予約、お役立ち情報
- 特徴: **電話が主力**。LINEは補助的な連絡手段の位置づけ
- コンテンツ: 看護技術解説、働き方コラム、マンガコンテンツ

**ナースロビーとの差**:
- レバウェルはLINEを「連絡手段」として使い、ナースロビーは「ファネルそのもの」として使っている
- レバウェルは登録後すぐに電話がかかってくる（口コミで「しつこい」と評判）
- ナースロビーの「電話なし・LINE完結」は明確な差別化

---

### 17. ナース人材バンク（ナース専科 転職）のLINEは？

**確認できた情報**:
- 登録後にLINE友だち追加案内が送られる（10秒で手続き完了）
- 担当キャリアパートナーとのLINEチャットが可能
- 求人紹介は電話・メール・LINEの選択制
- リッチメニュー: 基本的な案内（求人検索はWebサイト誘導）

**ナースロビーとの差**:
- ナース専科はLINEを「担当者との連絡ツール」として使用
- 求人検索はWebサイトで行い、LINEに結果は表示されない
- ナースロビーはLINE内で完結するマッチング体験を提供 → 優位

---

### 18. マイナビ看護師のLINEは？

**確認できた情報**:
- おすすめ求人・転職お役立ち情報を日々配信
- 「チャットで面談予約」機能あり（リッチメニューから）
- キャリアアドバイザーとのLINEコミュニケーション
- 主力はWebアプリ。LINEは通知+予約チャネル

**ナースロビーとの差**:
- マイナビはアプリ+Webが主力で、LINEは補助
- 「面談予約」をLINEリッチメニューに配置 → ナースロビーの「担当者に相談」に相当
- マイナビは規模の経済（求人数10万件超）で差をつける

---

### 19. 大手にあってナースロビーにないものは？

| 機能 | レバウェル | ナース専科 | マイナビ | ナースロビー |
|------|-----------|-----------|---------|------------|
| 専任担当者 | ○ | ○ | ○ | △（1人体制） |
| 求人数 | 15万件 | 20万件 | 10万件 | 2,936件 |
| 非公開求人 | ○ | ○ | ○ | ✕ |
| 面接同行 | ○ | △ | ○ | ✕ |
| 給与交渉代行 | ○ | ○ | ○ | △ |
| 入職後フォロー | ○ | ○ | ○ | ✕ |
| コンテンツ（コラム等） | ○ | ○ | ○ | △（FAQ 5問） |
| Webアプリ | ○ | ○ | ○ | ✕（LINE完結） |

**致命的に足りないもの**: 求人数。2,936件 vs 10-20万件は桁が違う。
**ただし**: 大手の求人数は「全国」であり、関東4都県×看護師に絞ると差は縮まる。また、ナースロビーはD1 facilitiesに24,488施設を持っており、施設情報の提供は可能。

---

### 20. ナースロビーにあって大手にないものは？

| 差別化要素 | 詳細 |
|-----------|------|
| **手数料10%** | 業界最安。病院側の負担が軽い → 内定率が高くなる可能性 |
| **完全LINE完結** | 電話なし。登録から求人閲覧まで全てLINE内 |
| **即時マッチング** | 5問タップで即結果。大手は登録→電話→ヒアリング→数日後に求人提案 |
| **AI活用** | AI相談、AI履歴書作成（大手はまだ人力中心） |
| **匿名性** | 名前を聞かない。個人情報最小限で求人が見られる |
| **24時間対応** | Botなので深夜でも求人検索可能 |
| **4段階AIフォールバック** | OpenAI→Claude→Gemini→Workers AIで応答途切れなし |

**最大の武器は「速さ」と「気軽さ」**。大手は「手厚いサポート」で勝負するが、そのサポートが「しつこい」と感じる層が一定数いる。ナースロビーはその層を狙える。

### 23. LINEブロック率を下げる最も効果的な機能は？

**ブロックされる主な理由**:
1. 登録直後に期待と違った（42%）
2. メッセージが多すぎる/しつこい（28%）
3. 有益な情報がない（18%）
4. 使い方がわからない（12%）

**最も効果的な機能**: **転職Q&A / お役立ちコンテンツ**

理由:
- 求人を探す気がない時でも「役に立つ」と感じさせられる
- 「年収相場を知りたい」「バレずに転職活動するには？」は看護師の日常的な関心事
- ブロックする前に「このアカウント、情報源として残しておこう」と思わせる
- 医療・介護人材転職のトライト社事例: 雑学・トリビア・リッチメニューコンテンツ中心の配信で高いエンゲージメントを維持

**次点**: リッチメニュー自体の存在感。リッチメニューがあるだけで「このアカウントはアクティブで機能がある」という印象を与え、ブロック率が下がるという調査結果がある。

---

### 24. handoff率を上げる最も効果的な機能は？

**現在のhandoff導線**:
```
5問回答 → マッチング結果5件表示 → 「この施設について聞く」or「直接相談する」→ handoff
```

**ボトルネック分析**:
- 5問回答まで到達しない人（離脱）→ 上段CTAの改善
- マッチング結果に満足しない人 → 0件撲滅は実装済み
- 「聞く」のハードルが高い → 「気軽さ」の演出不足

**最も効果的な機能**: **リッチメニューへの「担当者に相談」ボタン追加**

理由:
- 5問のフローをスキップして直接handoffできる
- 「もう決めてる。早く話したい」層を取りこぼさない
- matched/hearing状態のリッチメニューにも配置することで、全フェーズからhandoff可能に
- 実装コスト: 1-2時間（既存handoff=ok postbackを呼ぶだけ）

### 25. 月間1人の成約を達成するために最も重要な機能は？

**逆算**:
```
成約1件 ← 入職1.1件（90%） ← 内定2件（50%） ← 紹介2.5件（80%）
← 面談4件（60%） ← ヒアリング5件（80%） ← LINE登録6-7件（転職意向あり）
← LINE友だち追加25-50件（LINE登録のうち転職本気は15-25%）
← LP訪問250-500件（LINE登録率5-10%）
```

**結論: リッチメニューの機能改善ではなく、集客がボトルネック**。

現在LINE登録数は0人。リッチメニューを完璧にしても、ユーザーが来なければ意味がない。

ただしリッチメニューが影響するのは以下の変換率:
- LINE登録→ヒアリング完了率: 80%目標。現行5問フローで十分達成可能
- ヒアリング→handoff率: ここが最大のレバー。**「担当者に相談」ボタンが最も重要**

**月間1人の成約に最も重要な機能の優先順位**:

1. **「お仕事探しをスタート」の維持・最適化** — メインファネルの入口
2. **「担当者に相談」の追加** — handoff率を最大化するショートカット
3. **「転職Q&A」の追加** — ブロック防止 → ファネル内ユーザー数の維持
4. 「新着求人」— リピート訪問の動機付け
5. 「施設検索」— 特定ニーズへの対応
6. 「履歴書作成」— 差別化だが優先度は低い（成約直前フェーズ）

## 総合提案: リッチメニューv2 推奨構成

### Default状態（初回〜ヒアリング前）

```
┌─────────────────────────────────────┐
│     お仕事探しをスタート              │  上段（全幅）— 既存メインCTA
├───────────┬───────────┬─────────────┤
│ Q&A       │ NEW       │ 相談する    │  下段（3分割）
│ 転職の    │ 最新の    │ 担当者に    │
│ ギモン    │ 求人      │ 直接相談    │
└───────────┴───────────┴─────────────┘
```

**変更理由**:
- 「施設検索」→ 削除。エリア・条件検索は5問フローと重複。施設名検索のニーズはAI相談テキスト入力で対応可能
- 「履歴書作成」→ matched状態のメニューに移動。default段階では不要
- 「転職Q&A」→ 追加。迷い中ユーザーの受け皿+ブロック防止。既存FAQ5問を活用（工数2-3h）
- 「担当者に相談」→ 追加。handoff率最大化のショートカット（工数1-2h）
- 「新着求人」→ 維持。ラベルを「本日の新着」→「最新の求人」に変更（0件リスク回避）

### Matched状態（求人表示後）

```
┌─────────────────────────────────────┐
│     もっと求人を見る                  │  上段
├───────────┬───────────┬─────────────┤
│ 条件変更  │ 履歴書    │ 相談する    │  下段
│           │ AI作成    │ 担当者に    │
└───────────┴───────────┴─────────────┘
```

### Handoff状態（担当者対応中）

```
┌─────────────────────────────────────┐
│     担当者にメッセージを送る          │  上段
├───────────┬───────────┬─────────────┤
│ Q&A       │ 求人を    │ 面接準備    │  下段
│ 転職の    │ もう一度  │ ガイド      │
│ ギモン    │ 見る      │             │
└───────────┴───────────┴─────────────┘
```
