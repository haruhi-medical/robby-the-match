# LINE Bot welcomeメッセージ改善提案

作成日: 2026-04-06
作成者: 6人専門家チーム

---

## 1. 質問ステップの正確な整理

### フロー全体図

```
welcome → il_area（都道府県） → il_subarea（サブエリア） → il_facility_type（施設タイプ）
  → [病院の場合] il_department（診療科） → il_workstyle（働き方） → il_urgency（転職温度感） → matching_preview
  → [クリニックの場合] il_workstyle（働き方・2択に短縮） → il_urgency → matching_preview
  → [訪問/介護/こだわりなしの場合] il_workstyle（働き方・4択） → il_urgency → matching_preview
```

### 各ステップの質問内容

| # | phase | 質問 | 選択肢数 | スキップ条件 |
|---|-------|------|---------|------------|
| 1 | il_area | どのエリアで働きたいですか？ | 5（東京/神奈川/千葉/埼玉/その他） | なし |
| 2 | il_subarea | ○○のどのあたりが希望ですか？ | 3-6（県により異なる） | なし |
| 3 | il_facility_type | どんな職場が気になりますか？ | 7（急性期/回復期/慢性期/クリニック/訪問/介護/こだわりなし） | なし |
| 4 | il_department | 希望の診療科はありますか？ | 10（内科系〜こだわりなし） | **病院（急性期/回復期/慢性期）選択時のみ表示** |
| 5 | il_workstyle | 希望の働き方は？ | 4（日勤/夜勤あり/パート/夜勤専従）※クリニック時は2択 | なし |
| 6 | il_urgency | 今の転職への気持ちは？ | 3（すぐにでも/いい求人があれば/情報収集） | なし |

### 最短・最長ステップ数

- **最短: 5ステップ**（クリニック/訪問/介護/こだわりなし選択時 → il_departmentスキップ）
  - il_area → il_subarea → il_facility_type → il_workstyle → il_urgency
- **最長: 6ステップ**（病院選択時 → il_department追加）
  - il_area → il_subarea → il_facility_type → il_department → il_workstyle → il_urgency

※診断引き継ぎ（shindan）の場合は0ステップ（LP側で3問回答済み → 即matching）
※地域ページ（area_page）の場合はエリア確定済みで働き方から開始（2ステップ短縮）

## 3. 現状の問題点

6人パネル分析結果:

| 専門家 | 指摘 |
|--------|------|
| **コピーライター** | 8パターンの保守コストが高い。「3つだけ」は嘘（実際は5-6ステップ）。フックが弱い。入口で3-4択のQuick Replyがあると迷って離脱する |
| **UXデザイナー** | 入口の分岐が多すぎてユーザー体験が不統一。「何かお手伝いできますか？」は受動的すぎる |
| **行動心理学者** | 不安除去（電話なし/ブロックOK）は良いが、具体的メリット（候補数）が弱い。「3つだけ」の数字は信頼を作るが、事実と異なるのは逆効果 |
| **看護師経験者** | 夜勤明けで疲れてる時に長文は読めない。「転職サポートです」は他社と同じ。差別化が見えない |
| **マーケター** | Quick Reply1択にすれば迷わない。入口を1本化すれば全A/Bテストが1箇所で済む |
| **法務** | 「完全無料」は問題なし。許可番号は初回メッセージでなくリッチメニューで十分 |

### 重要な事実確認
- 質問数は**5問（最短）〜6問（病院選択時）** → 「3つだけ」は不正確
- 全てタップ操作のみ（テキスト入力不要）
- 24,488件の施設DB（動的にカウント表示される）

## 4. 新welcomeメッセージ 3案

### 条件の再確認
- 全入口共通の1本（shindan/area_page除く。これらはセッション引き継ぎが機能的に必要）
- Quick Reply: 「求人を見てみる」1つだけ
- 5-8行
- キャラ紹介なし

---

### 案A: 「数字で止める」型

```
神奈川・東京・千葉・埼玉の
看護師求人、24,000件以上。

5つの質問をタップするだけで
あなたの条件に合う求人だけ届きます。

名前も電話番号もいりません。
合わなければいつでもブロックOK。
```
Quick Reply: `求人を見てみる`

**ミサキの反応予測:**
夜勤明けの朝、Instagramの広告からLINE追加。「24,000件」の数字で目が止まる。「タップするだけ」で楽そう。「名前も電話番号もいりません」で転職サイトのトラウマ（しつこい電話）が和らぐ。「ブロックOK」で最後の心理的ハードルが消える。ボタンが1つしかないので迷わずタップ。
**弱点:** 数字が大きすぎて「本当？」と疑う可能性。施設数であり求人数ではない点のミスリードリスク。

---

### 案B: 「不安除去ファースト」型

```
電話なし。名前もいりません。
全部LINEのタップだけで完結します。

5つ選ぶだけで、
あなたの条件に合う求人をお見せします。

合わなければブロックでOK。
まずは気軽にどうぞ。
```
Quick Reply: `求人を見てみる`

**ミサキの反応予測:**
1行目「電話なし。名前もいりません。」で一気に安心する。以前の転職サイトで名前と電話番号を入れた瞬間から鳴り止まなかった電話のトラウマが、最初の1行で解消される。「タップだけ」で面倒くささがない。「ブロックでOK」は退路の保証。夜勤明けの疲れた頭でも、不安がないからタップできる。
**弱点:** 差別化が弱い。「何の求人？」が伝わりにくい（看護師専門であることが明示されていない）。

---

### 案C: 「行動を最小化」型

```
看護師さんの求人、
5つタップするだけで届きます。

電話も名前も不要。
合わなければブロックでOK。
```
Quick Reply: `求人を見てみる`

**ミサキの反応予測:**
4行しかない。夜勤明けでも3秒で全部読める。「5つタップするだけ」が行動コストの低さを伝える。余計な情報がないから迷わない。ただし、あまりに短いので「怪しい？」と思うかもしれない。信頼の根拠（件数・無料）がないため、広告経由の初接触では弱い可能性。
**弱点:** 短すぎて信頼構築が不十分。初接触で「なぜここを信じるのか」の根拠がない。

## 5. 推奨案と理由

### 推奨: 案A（「数字で止める」型）

**理由:**

1. **フック力**: 「24,000件以上」は転職サイトのメール「あなたに合う求人3件」と桁が違う。夜勤明けでも数字は目に入る

2. **正確性**: 「5つの質問をタップするだけ」は事実（最短5ステップ、最長6ステップだが病院選択者は診療科が1問増えるだけ。5問ベースで案内するのが適切）

3. **不安除去の完備**: 「名前も電話番号もいりません」「ブロックOK」の2段構え。ミサキの転職サイトトラウマを直接解消

4. **汎用性**: 人間関係で辞めたい人にも、給与を上げたい人にも、夜勤を減らしたい人にも刺さる。転職理由に依存しない

5. **Quick Reply 1択**: 迷う余地がない。タップ率が最大化される

6. **保守コスト**: 1本統一なので変更・A/Bテストが容易

### 実装上の注意

- `shindan`（診断引き継ぎ）と`area_page`（地域ページ経由）は現行のまま維持（セッション引き継ぎが機能的に必要なため）
- それ以外の6パターン（meta_ad / hero / sticky / bottom / blog / salary_check / デフォルト）を案Aに統一
- 24,000件の数字は`countCandidatesD1({})`の動的取得値に置き換えること（ハードコード禁止）
- 「5つの質問」は最短ステップ数に基づく表現。病院選択で+1問になるが、診療科は「質問」というよりフィルタの感覚なので許容範囲

### 数字のミスリードリスクへの対処

案Aの「24,000件以上」は施設数（24,488施設）であり求人数ではない。ただしil_areaステップで「○○件の医療機関の中から」と表示されるため、文脈として「医療機関数」であることは自然に伝わる。welcomeメッセージでは「求人」という表現を使い、il_areaで「医療機関」に切り替わる設計。

もし正確性を優先するなら:
```
神奈川・東京・千葉・埼玉
24,000件以上の医療機関から探せます。
```
に変更する。
